# 需求路由（demand-router）

把口语需求压成“9 个 oh-doc 一级系统”的分包清单，与 `SKILL.md` 的 Workflow 对齐。

## 目标

把原始需求稳定拆成“可分派给子系统的结构化文档更新任务”，而不是写成一篇混合描述或直接跳进实现。

## 输入解析顺序

1. `玩家目标`
   - 玩家这次想达成什么结果。
2. `输入动作`
   - 玩家具体做了什么操作。
3. `画面反馈`
   - 玩家在屏幕上必须看到什么变化。
4. `状态承诺`
   - 这次需求隐含了哪些地图状态、实体状态、工作状态、行为状态、需求状态或 UI 状态变化。
5. `边界条件`
   - 哪些部分可以 fake/stub，哪些必须真实。

## 9 系统拆分规则

满足以下任一条件时，应拆出独立系统：

- 该部分有独立的玩家可见结果。
- 该部分需要独立测试层级。
- 该部分拥有不同的 fake/stub 边界。
- 该部分的修改会改变另一个系统的接口或依赖。

## 常见系统分类

- `UI系统`
  - 处理菜单、工具栏、模式提示、状态展示和地图叠加反馈。
- `交互系统`
  - 处理输入会话、模式切换、交互意图生成和预览反馈协调。
- `地图系统`
  - 处理格子、覆盖格集合、占用、区域和空间查询。
- `实体系统`
  - 处理实体原型、生命周期、关系一致性和只读投影。
- `工作系统`
  - 处理工作生成、工作领取、工作结算和复合工作链。
- `建筑系统`
  - 处理建筑规格、蓝图放置、施工进度和建成后结果。
- `时间系统`
  - 处理世界时钟、昼夜切换、时间推进和时间事件。
- `行为系统`
  - 处理决策输入、行为评分、状态机和行为执行协调。
- `需求系统`
  - 处理饱食度、精力值、阈值规则、恢复和需求建议。

## 边界判断准则

- `UI系统` 与 `交互系统` 必须分开。
  - 菜单、HUD、状态展示、地图叠加反馈属于 `UI系统`。
  - 输入解释、模式注册、拖拽会话、命令生成属于 `交互系统`。
- `建筑系统`、`工作系统`、`实体系统` 必须按职责分开。
  - 蓝图规格与放置属于 `建筑系统`。
  - 工作单与执行链属于 `工作系统`。
  - 蓝图、建筑、树木、物资、小人等对象真相属于 `实体系统`。
- `行为系统` 与 `需求系统` 必须分开。
  - 需求压力计算、阈值、恢复属于 `需求系统`。
  - 是否打断工作、切换到休息或进食属于 `行为系统`。
- 不要把需求重新折叠进旧的 `world-core`、`task-planning`、`scene-hud`、`selection-ui`。

## 输出格式

拆分结果应至少包含：

- `影响系统`
- `依赖系统`
- `每个系统负责的玩家可见结果`
- `每个系统要同步的 oh-doc`
- `每个系统的默认验证层级`
- `共享 fake/stub 清单`

## 典型路由样例

### 墙体蓝图绘制

- `交互系统`
  - 接收蓝图笔刷拖拽，输出覆盖格集合与提交命令。
- `地图系统`
  - 校验覆盖格合法性、冲突和边界。
- `建筑系统`
  - 根据路径创建墙体蓝图并维护建筑规格。
- `工作系统`
  - 为墙体蓝图生成建造工作并编排后续施工链。
- `实体系统`
  - 维护蓝图实体、建成后的建筑实体及相关关系。
- `UI系统`
  - 展示当前模式提示、蓝图虚影和地图叠加反馈。

### 夜晚休息与精力恢复

- `时间系统`
  - 触发夜晚时段与时间推进事件。
- `行为系统`
  - 响应夜晚事件并驱动小人切换到休息行为。
- `需求系统`
  - 结算精力恢复、疲劳阶段变化和需求建议。
- `实体系统`
  - 提供床铺归属、小人状态和相关只读实体快照。
- `UI系统`
  - 展示时间变化、当前行为标签和需求恢复反馈。

## 执行摘要

1. **提炼**：玩家目标 → 输入动作 → 画面反馈 → 状态承诺 → 边界与 fake/stub。
2. **拆分**：按独立可见结果 / 测试层级 / 假实现边界划分 9 个一级系统。
3. **查表**：打开 `system-registry.md`，确认每个系统的 `oh-gen-doc`、`oh-code-design`、`oh-acceptance` 和默认验证层级。
4. **分派**：每系统单独收敛文档更新建议包，约定见 `subagent-contract.md`。
5. **汇总**：主控路由单、文档更新顺序、测试入口建议与 fake-to-real 顺序；测试策略见 `skill-tdd.md`。
