# 需求路由（demand-router）

把口语需求压成可执行分包流程时的检查清单，与 `SKILL.md` 的 Workflow 对齐。

## 目标

把原始需求稳定拆成“可分派给子系统”的工作单元，而不是写成一篇混合描述。

## 输入解析顺序

1. `玩家目标`
   - 玩家这次想达成什么结果。
2. `输入动作`
   - 玩家具体做了什么操作。
3. `画面反馈`
   - 玩家在屏幕上必须看到什么变化。
4. `状态承诺`
   - 这次需求隐含了哪些世界状态、角色状态、UI 状态或任务状态变化。
5. `边界条件`
   - 哪些部分可以 fake/stub，哪些必须真实。

## 系统拆分规则

满足以下任一条件时，应拆出独立系统：

- 该部分有独立的玩家可见结果。
- 该部分需要独立测试层级。
- 该部分拥有不同的假实现边界。
- 该部分的修改会改变另一个系统的接口或依赖。

## 常见系统分类

- `selection-ui`
  处理选中、焦点、可交互高亮和对应反馈。
- `scene-hud`
  处理 HUD、面板、按钮、状态卡和信息展示。
- `time-of-day`
  处理世界时间推进、时间格式化和昼夜视觉调色板。
- `pawn-state`
  处理角色状态、属性、需求和可暴露给 UI 的状态模型。
- `task-planning`
  处理目标评估、任务选择、计划与工作分配。
- `world-grid`
  处理地图尺寸、格子坐标、邻格、可行走边界和占用快照。

## 输出格式

拆分结果应至少包含：

- `目标系统`
- `依赖系统`
- `每个目标系统负责的玩家可见结果`
- `每个目标系统的默认测试层级`
- `共享 fake/stub 清单`

## 决策准则

- 单系统可完成的需求，不要过度拆分。
- 涉及 UI 与领域状态时，优先拆出 UI 系统和领域系统，而不是混写。
- 如果底层规则未定但交互必须先验证，优先把 UI 系统列为目标系统，领域系统列为后续依赖。

## 执行摘要

1. **提炼**：玩家目标 → 输入动作 → 画面反馈 → 状态承诺 → 边界与 fake/stub。
2. **拆分**：按独立可见结果 / 测试层级 / 假实现边界划分子系统。
3. **查表**：打开 `system-registry.md`，确认标准文档路径、aidoc 目录与默认 failing test。
4. **分派**：每系统单独收敛规格，约定见 `subagent-contract.md`。
5. **汇总**：主控需求单、集成文档、TDD 顺序与 fake-to-real；测试策略见 `skill-tdd.md`。
