# 任务工作流

Agent 默认工作流：

1. 阅读 `Agent.md` 以及 `.agent/` 下的相关文件。
2. 在提出架构改动前先检查本地代码与文档。
3. 对于新增玩法、改交互、扩系统、补规则的需求，先走 `.agent/skills/route-demand/SKILL.md`。
4. 先生成主控需求单、子系统 aidoc 与集成文档，再进入实现。
5. 以小步提交的方式实现。
6. 为确定性行为补充或更新测试。
7. 准备推送远端前，必须走 `.agent/skills/push-with-aidoc/SKILL.md`，检查受影响系统的 aidoc、索引与受控提交格式。
8. 当初始化方式或协作流程变化时，同步更新面向人的文档。

功能开发工作流：

1. 明确玩法意图。
2. 若涉及多个系统，先用 `route-demand` 将需求拆成子系统 aidoc。
3. 明确模拟实体与状态迁移。
4. 明确角色 AI 如何评估目标并选择工作。
5. 对 UI-first 原型，允许先用 fake/stub 跑通交互测试，再反推底层逻辑。
6. 在 Phaser 场景中暴露对应行为。
7. 通过测试验证，并补上一段简短文档说明。
8. 推送前通过 `push-with-aidoc` 回填受影响系统的 `docs/ai/index/`、系统 README 与 routed aidoc。

缺陷修复工作流：

1. 复现或隔离问题。
2. 确认问题发生在模拟层还是表现层。
3. 在最小且正确的层级完成修复。
4. 条件允许时加入回归测试。
5. 记录任何值得说明的规则变化。
6. 推送前通过 `push-with-aidoc` 校验本次修复是否需要补系统 aidoc 或集成文档。
