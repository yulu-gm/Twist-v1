# 仓库规则

仓库结构意图：

- `src/`：游戏运行时代码
- `src/game/`：核心模拟、世界状态、实体与系统
- `src/scenes/`：Phaser 场景与表现层编排
- `tests/`：自动化测试
- `data/`：静态游戏数据、JSON 配置与数值表
- `tools/`：校验脚本、内容流水线与开发工具
- `docs/human/`：面向开发者、设计师与玩家的文档
- `docs/ai/`：架构说明、Agent 指南与机器导向规格

规则：

- 除非有明确理由，否则不要把纯渲染逻辑与模拟状态更新混在一起。
- 新的玩法功能应先在 `docs/ai/` 中以轻量规格说明开头。
- 新的内容格式在适用时必须在 `data/` 下附带示例文件。
- 测试应优先覆盖确定性系统，例如工具逻辑、规划器、状态迁移与规则评估。
- 避免让 Phaser API 与领域逻辑形成深度耦合。

分支与变更管理：

- 每次改动都应聚焦，并且目标明确。
- 如果文件已经被用户修改，必须保留用户已有工作。
- 文档更新应优先采用补充式说明，而不是无说明重写。
