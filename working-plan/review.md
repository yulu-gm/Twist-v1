  Review：当前实现 vs oh-code-design 规范

  一、架构层面的核心问题

  1. 缺少分层架构（最严重）

  设计规范为每个子系统定义了 4 层架构（模型层 → 规则层 → 编排层 → 投影层），但当前实现几乎全部是 扁平函数 +
  单文件巨型模块：

  - world-core.ts（919 行）把实体目录、生命周期规则、占用管理、工作调度、建筑结算、时间推进全部混在一个文件里
  - GameScene.ts（794 行）承担了场景编排、输入处理、UI 同步、模拟驱动、渲染协调等全部职责
  - 没有独立的"规则层"和"投影层"，规则散落在各处

  2. 子系统边界不存在

  设计规范定义了 9 个独立子系统，各自有明确的模块划分和接口边界。当前实现：

  ┌────────────┬──────────────────────────────────────────────┬──────────────────────────────────────────────────────┐
  │ 设计子系统 │                   当前实现                   │                         问题                         │
  ├────────────┼──────────────────────────────────────────────┼──────────────────────────────────────────────────────┤
  │ 实体系统   │ 混在 world-core.ts 里                        │ 无独立实体目录、无原型定义、无生命周期规则模块       │
  ├────────────┼──────────────────────────────────────────────┼──────────────────────────────────────────────────────┤
  │ 地图系统   │ world-grid.ts 只有坐标工具                   │ 缺少占用管理器、区域管理器、选区解析器               │
  ├────────────┼──────────────────────────────────────────────┼──────────────────────────────────────────────────────┤
  │ 时间系统   │ time-of-day.ts 较完整                        │ 缺少时间事件总线，事件只在 world-core 内部处理       │
  ├────────────┼──────────────────────────────────────────────┼──────────────────────────────────────────────────────┤
  │ 工作系统   │ 混在 world-core.ts 里                        │ 无工作目录、无工作生成器、无工作编排器、无工作结算器 │
  ├────────────┼──────────────────────────────────────────────┼──────────────────────────────────────────────────────┤
  │ 建筑系统   │ 混在 world-core.ts + apply-domain-command.ts │ 无建筑规格目录、无蓝图管理器、无归属规则器           │
  ├────────────┼──────────────────────────────────────────────┼──────────────────────────────────────────────────────┤
  │ 行为系统   │ sim-loop.ts + goal-driven-planning.ts        │ 无行为状态机、无行为上下文汇总器、无行为执行器       │
  ├────────────┼──────────────────────────────────────────────┼──────────────────────────────────────────────────────┤
  │ 需求系统   │ 散在 pawn-state.ts + need-signals.ts         │ 无需求档案、无阈值规则集、无需求演化引擎             │
  ├────────────┼──────────────────────────────────────────────┼──────────────────────────────────────────────────────┤
  │ 交互系统   │ player/ 目录多个文件                         │ 无模式注册表、无统一会话管理                         │
  ├────────────┼──────────────────────────────────────────────┼──────────────────────────────────────────────────────┤
  │ UI 系统    │ hud-manager.ts 单文件                        │ 无菜单模型、无状态展示模型分离                       │
  └────────────┴──────────────────────────────────────────────┴──────────────────────────────────────────────────────┘

  3. 实体模型不完整

  设计规范定义了 6 种实体（小人、物资、树木、蓝图、建筑、区域），当前实现：
  - 小人（Pawn） 与世界实体完全分离（PawnState vs WorldEntitySnapshot），不在同一实体目录内
  - 物资实体 不存在（只有 mock 的 MOCK_SCATTERED_GROUND_ITEMS）
  - 树木实体 不存在（障碍物没有伐木标记等字段）
  - 区域实体 不存在（无存储区概念）
  - 实体缺少关键字段：携带物标识、饱食度/精力值（在 PawnState 而非实体上）、物资类型、容器关系等

  5. 小人 AI 与世界内核割裂

  - PawnState 在 pawn-state.ts 中独立管理，与 WorldCore 的实体系统完全隔离
  - 小人的需求（hunger/rest/recreation）用 0-100
  反向刻度（越高越急），与设计中"饱食度/精力值"（正向值，越低越急）语义相反
  - sim-loop.ts 中的 AI tick 只操作 PawnState[] 和 ReservationSnapshot，不读写 WorldCore
  - 小人移动只操作逻辑格+插值，不更新世界占用

  6. 接口契约（S0）不完整

  s0-contract.ts 只定义了 DomainCommand 和 MockWorldSubmitResult，缺少设计中约定的：
  - 实体标识与只读实体快照字段
  - 覆盖格集合、占用冲突返回值
  - 时间快照与时间事件格式（作为契约）
  - 工作单、工作步骤、工作锁定与完成回报格式
  - 需求到行为的优先级/紧急度/打断信号

  ---
  二、重构设计计划

  大体方向如下：

  Phase 0：统一实体模型（基础，其他 Phase 依赖）

  - 将 PawnState 纳入统一实体系统，建立 src/game/entity/ 目录
  - 定义 6 种实体原型（小人、物资、树木、蓝图、建筑、区域）
  - 实现实体目录（EntityRegistry）和生命周期规则

  Phase 1：拆分子系统目录结构

  - 按 9 大子系统建立独立目录：entity/、map/、time/、work/、building/、behavior/、need/、interaction/、ui/
  - 每个子系统内部按设计的 4 层架构组织

  Phase 2：实现 A 线缺失模块

  - 地图系统：占用管理器、区域管理器
  - 工作系统：工作目录、工作生成器、工作编排器、工作结算器
  - 建筑系统：建筑规格目录、蓝图管理器、建成结算器、归属规则器
  - 时间系统：时间事件总线

  Phase 3：实现 B 线缺失模块

  - 行为系统：行为状态机、行为上下文汇总器、行动评分器
  - 需求系统：需求档案、阈值规则集、需求演化引擎、满足结算器
  - 交互系统：模式注册表、统一会话管理
  - UI 系统：菜单模型、状态展示模型

  Phase 4：打通关键链路

  - 伐木→木头→拾取→搬运
  - 蓝图→建造→建筑→床铺分配
  - 需求→打断→恢复→重评
  - 夜晚→休息→唤醒

  Phase 5：瘦身 GameScene

  - 将 GameScene 拆为纯渲染编排器，领域逻辑全部下沉到子系统
