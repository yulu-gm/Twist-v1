# oh-acceptance 验收场景覆盖缺口清单

## 判定标准

本清单按以下标准判断“已覆盖”：

- 必须经过正常游戏流程：`GameScene` 运行、玩家选择工具或菜单、真实指针输入或等价 UI 操作、世界状态变化、玩家可见反馈。
- 不接受仅无头通过：只跑 `headless` 编排、只看领域状态、直接 `submit` 命令、直接改世界或 Pawn 状态、跳过渲染层与玩家操作，均不算通过。
- 若场景只在浏览器右下角 `scenario` 面板中“装载初始状态”，但没有从玩家输入开始执行，也不算通过。

## 总览

- 按上述标准，当前完整覆盖数：`0`
- 当前主要问题不是“少几个测试”，而是验收入口整体偏向 `headless` 和状态注入。
- 现有 `scenario` 体系更接近“领域回归场景”而不是“玩家验收场景”。

## 全局结构性缺口

1. 缺少真正的玩家验收运行器。
原因：仓库只有 `vitest`、`tests/headless/*`、少量 `happy-dom` 组件测试，没有浏览器 E2E 或 Phaser 场景驱动测试。

2. `scenario` 会在装载阶段直接帮测试做动作。
原因：`domainCommandsAfterHydrate`、`playerSelectionAfterHydrate`、`claimConstructBlueprintAsPawnName` 会在 `hydrate/load` 时自动提交命令或自动认领工单，绕过玩家输入链路。

3. 浏览器中的 `scenario` 面板本质是“载入状态”，不是“执行验收”。
原因：`GameScene.applyHeadlessScenarioDefinition` 只是把定义写入当前世界并重绘，没有自动验证，也没有要求从菜单/鼠标交互重新走流程。

4. UI 渲染结果几乎没有自动验证。
原因：现有 UI 测试只覆盖 DOM 标记、菜单状态和文字映射，没有驱动 Phaser 指针事件，也没有断言选区框、虚影、进度条、层级遮挡、暂停冻结等运行态表现。

## 缺口清单

### 交互系统

`INTERACT-001 使用选区工具标记物资可拾取`
当前：只有 `haul-mark-pickup` 的 headless/领域验证，且场景默认通过 `domainCommandsAfterHydrate` 直接下发 `haul`。
缺在哪：没有真实玩家拖拽选区，没有验证拖拽中的半透明选区框，也没有验证释放后地图上标记图标的可见表现。
为什么缺：当前测试直接构造或提交命令，跳过 `GameSceneFloorInteraction` 的 `pointerdown -> pointermove -> pointerup` 路径。

`INTERACT-002 使用蓝图笔刷连续绘制墙体`
当前：`build-wall-flow` 和相关 headless 测试能证明 `build + brush-stroke` 最终生成蓝图和工单。
缺在哪：没有在 `GameScene` 里通过菜单选“建造 -> 墙 -> 木墙”后再真实拖拽，没有验证拖拽过程中的墙体虚影实时显示。
为什么缺：当前 `scenario` 在 hydrate 后直接执行 `playerSelectionAfterHydrate`，测试主要看世界状态，不看渲染中的笔刷反馈。

`INTERACT-003 单点点击放置家具`
当前：`build-bed-flow` 的 headless 测试能证明 `build + single-cell` 可生成床蓝图和工单。
缺在哪：没有真实点击放置，没有验证鼠标悬停虚影，也没有验证点击后的建造提示在玩家视角是否成立。
为什么缺：测试直接调用 `commitPlayerSelection` 或预置蓝图，未驱动 Phaser 场景输入。

`INTERACT-004 无效交互模式下的输入屏蔽`
当前：无对应玩家验收场景。
缺在哪：没有验证默认观察态下点击/拖拽不会生成命令，也没有验证不会出现选区框、虚影或标记。
为什么缺：现有测试集中在“有工具、有效命令”的路径，没有覆盖空模式输入屏蔽。

### 地图系统

`MAP-001 第一天地图初始状态生成`
当前：`story-1-day-one` 和 `map-initial-state` 只验证初始实体存在，浏览器侧依赖场景装载或 bootstrap。
缺在哪：缺少以玩家启动游戏后的实机场景渲染验收，未自动验证“小人、树木、散落物资实际被渲染出来”。
为什么缺：自动测试只看实体存在，不看 Phaser 画面结果。

`MAP-002 合法选区创建存储区`
当前：`zone-create.test.ts` 和 `story-1-day-one` 能验证 `zone_create` 后确实出现 zone 实体。
缺在哪：没有真实玩家处于存储区模式后拖拽框选，也没有验证地图上存储区边界/底色的表现。
为什么缺：测试直接 `submit(zone_create)`，没有走选区输入和渲染层。

`MAP-003 建筑放置时的占用冲突拦截`
当前：有领域拒绝逻辑和冲突相关测试。
缺在哪：没有玩家放置蓝图时看到红色不可放置虚影的验收。
为什么缺：现有验证只证明世界端会拒绝，不证明玩家输入时的预览反馈成立。

`MAP-004 越界选区的安全过滤`
当前：`GameSceneFloorInteraction.pointerCell(..., true)` 有裁剪实现，但无验收测试。
缺在哪：没有验证玩家拖拽越界时只高亮合法区域，也没有验证越界部分被裁剪后的视觉结果。
为什么缺：当前没有 Phaser 级拖拽测试。

### 工作系统

`WORK-001 伐木工作的生成与完整链路执行`
当前：`chop-tree-full-flow`、`chop-haul-full-chain`、`story-1-day-one` 可证明伐木到产木、再到搬运的领域链路。
缺在哪：没有真实玩家使用伐木工具框选树木，没有验证树木标记 UI、小人伐木读条、树消失与木头模型出现的完整玩家可见流程。
为什么缺：场景通过 `domainCommandsAfterHydrate` 或直接 `submit(lumber)` 启动流程，渲染侧只靠人工观察，无自动验收。

`WORK-002 散落物资的自动拾取与搬运`
当前：`haul-mark-pickup`、`chop-haul-full-chain`、`story-1-day-one` 可证明 pick-up / haul-to-zone 领域链路。
缺在哪：没有真实玩家框选物资标记为可拾取，没有验证“地上消失、显示在小人手中或背包 UI、存储区重新出现”的表现。
为什么缺：自动测试只检查容器状态和工单状态。

`WORK-003 工作目标在执行前消失`
当前：有 `stale-work-cleanup` 类 headless/领域测试。
缺在哪：没有玩家视角验证“小人中途停下、任务 UI 移除”。
为什么缺：现有测试只关心工作状态回收，不关心表现层。

`WORK-004 多个小人争抢同一工作目标的互斥锁定`
当前：有 work scheduler / registry / multi-pawn 相关领域测试。
缺在哪：没有玩家验收“只有一个小人前往蓝图，其他小人忽略”的画面级确认。
为什么缺：没有多 Pawn 行为的场景级可视验收。

### 建筑系统

`BUILD-001 基础墙体蓝图放置与建造`
当前：`build-wall-flow`、`story-1-day-one` 能证明墙蓝图最终可落成。
缺在哪：正常流程要求玩家先通过菜单进入墙笔刷模式，再拖拽生成蓝图，再由小人建造；当前场景在加载时已替玩家提交甚至自动认领。
为什么缺：`build-wall-flow.scenario.ts` 直接使用 `playerSelectionAfterHydrate` 和 `claimConstructBlueprintAsPawnName`。

`BUILD-002 木床建造完成后的自动分配`
当前：`bed-auto-assign`、`story-1-day-one` 能证明床落成后 `ownerPawnId` 会被填充。
缺在哪：没有玩家放置床蓝图再观察“选中床铺 UI 显示所有者”的完整验收。
为什么缺：当前以世界 `restSpots` / ownership 断言为主，UI 所有者显示没有对应玩家测试。

`BUILD-003 非法位置的蓝图放置拦截`
当前：有世界侧冲突拒绝逻辑。
缺在哪：没有验证红色或不可放置虚影，也没有验证点击/拖拽后的失败反馈。
为什么缺：现有测试只验证提交结果，不验证预放置表现。

`BUILD-004 床铺数量多于无床小人时的分配`
当前：`bed-auto-assign.test.ts` 能验证多余床保持无 owner。
缺在哪：没有玩家验收“选中该床时 UI 显示无所有者”。
为什么缺：现有覆盖只在领域层断言 `ownerPawnId === undefined`。

### 实体系统

`ENTITY-001 树木被伐木完成转为木头物资`
当前：`chop-tree-full-flow`、`story-1-day-one` 可证明 tree 消失、resource 出现。
缺在哪：没有玩家视角验证树模型消失、木头模型出现。
为什么缺：自动测试不读渲染层。

`ENTITY-002 物资被拾取、携带与放下`
当前：`haul-mark-pickup`、`chop-haul-full-chain` 等能证明 container 状态变化。
缺在哪：没有玩家视角验证“物资附着到小人或 UI 背包”以及“放下时出现在存储区地块”。
为什么缺：当前只测实体容器字段，不测表现。

`ENTITY-003 蓝图建造完成转为建筑实体`
当前：`build-wall-flow`、`build-bed-flow` 可证明蓝图最终转建筑。
缺在哪：没有验证“半透明蓝图虚影消失并替换为真实模型”的渲染过程。
为什么缺：自动测试只看实体种类切换。

`ENTITY-004 实体归属关系的一致性保护`
当前：有关系一致性与领域规则测试。
缺在哪：没有玩家验收“视觉维持现状，不出现分身或瞬移”。
为什么缺：这类异常路径目前只在领域层保护，没有表现层验收。

### 需求系统

`NEED-001 饱食度随时间下降并触发进食`
当前：`pawn-eats-when-hungry` 只断言当前目标转为 `eat`。
缺在哪：没有验证需求条逐渐下降、变色警告、完成进食后回满。
为什么缺：场景只检查 AI goal，不检查 UI 条和提示图标。

`NEED-002 夜晚疲劳触发睡眠与精力恢复`
当前：`pawn-sleeps-when-tired`、`night-forces-sleep`、`story-1-day-one` 只证明目标会转向 `sleep`。
缺在哪：没有验证床上睡眠表现、精力条上涨、夜晚玩家可见流程。
为什么缺：测试只验证行为目标，不验证面板和动画。

`NEED-003 进食行为被强行中断`
当前：只有相近的 `need-interrupt-during-work`，但它是“工作被饥饿打断”，不是“进食被外力打断”。
缺在哪：目标场景要求验证进食读条中断后仅部分恢复饱食，这条没有对应场景。
为什么缺：现有实现和测试聚焦 hunger interrupt work，不是 interrupt eating。

`NEED-004 极度饥饿/疲劳的惩罚边界`
当前：有部分数值下限/行为单测，但没有玩家验收场景。
缺在哪：没有验证需求条清空闪烁红色，也没有验证极弱动画/昏倒等表现。
为什么缺：表现层要求未进入自动验收体系。

### 行为系统

`BEHAVIOR-001 空闲小人的漫无目的散步`
当前：有 `multi-pawn-colony` 和部分行为单测，但没有专门玩家验收。
缺在哪：没有明确验证“无工作、无紧迫需求时随机走动”以及 UI 显示空闲/散步。
为什么缺：现有多人场景只是宽泛冒烟，不是针对该验收点的玩家场景。

`BEHAVIOR-002 空闲状态转为执行工作`
当前：领域和工单系统有相关覆盖。
缺在哪：没有玩家验收“先散步，再收到工作后立刻转向目标”的可见行为。
为什么缺：缺少从场景演出角度组织的行为测试。

`BEHAVIOR-003 工作被紧急生理需求打断`
当前：`need-interrupt-during-work` 和 `story-1-day-one` 可证明工作会被释放并转向进食。
缺在哪：没有自动验证“小人中断动作和读条、UI 状态面板切为寻找食物”的完整玩家表现。
为什么缺：现有测试只看工单状态和 `currentGoal`。

`BEHAVIOR-004 无可用资源时的需求打断处理`
当前：无对应玩家验收场景。
缺在哪：没有验证“无床/无食物时的退化表现”和 UI 文案。
为什么缺：当前测试资源大多是“有目标可去”的 happy path。

### 时间系统

`TIME-001 游戏时间的实时推进与昼夜切换`
当前：有时间推进领域测试。
缺在哪：没有玩家验收 UI 时钟走动和环境光照平滑切换。
为什么缺：headless 无渲染，组件测试也不驱动 Phaser 场景。

`TIME-002 跨天归一化处理`
当前：有时间领域测试。
缺在哪：没有玩家验收日期 UI 切到第 2 天且环境过渡回白天。
为什么缺：无场景级时间表现测试。

`TIME-003 游戏暂停时的时间冻结`
当前：领域层能验证 paused 时 delta 为 0。
缺在哪：没有玩家点击暂停按钮后的完整验收，没有验证小人动作、进度条、时钟都冻结在当前帧。
为什么缺：缺少 `GameScene` 运行态冻结测试。

`TIME-004 极端帧间隔下的时间上限保护`
当前：部分时间领域逻辑可测。
缺在哪：没有玩家验收“恢复后只推进安全距离”的可见结果。
为什么缺：场景层没有模拟大帧间隔并观察 UI/角色变化的测试。

### UI 系统

`UI-001 主菜单层级展开与模式切换`
当前：`scene-hud-markup.test.ts` 能验证菜单 DOM 和 build submenu 基本展开、build 子工具状态映射。
缺在哪：没有把 HUD 菜单点击与 `GameScene` 的真实交互模式切换连起来，也没有验证状态展示区在运行态显示对应模式提示。
为什么缺：现有测试停留在 `HudManager` / `menu-model` 层，不连接 Phaser 场景。

`UI-002 地图叠加反馈层展示工作进度`
当前：`pawn-renderer` 实现了工作进度条显示逻辑。
缺在哪：没有测试驱动小人进入耗时工作并验证进度条渲染、平滑填充、完成后消失。
为什么缺：当前没有渲染层断言。

`UI-003 小人状态与需求面板的实时同步`
当前：`scene-hud-markup.test.ts` 只验证几种文本映射，如 `伐木中`、`搬运物资`、`进食中`、`空闲`。
缺在哪：没有在真实运行中验证需求值、警戒色、行为文本实时同步。
为什么缺：这是静态/组件级测试，不是玩家流程验收。

`UI-004 多层地图反馈的视觉清晰度`
当前：各类 renderer 已分配 depth。
缺在哪：没有验证选区框、蓝图虚影、标记图标、存储区边界同时出现时的层级清晰度。
为什么缺：当前没有多层反馈的场景级渲染比较。

部分覆盖：`INTERACT-002`、`BUILD-001`、`ENTITY-003`

- `build-bed-flow`
部分覆盖：`INTERACT-003`、`BUILD-002`、`ENTITY-003`
缺点：主场景直接预置蓝图并自动认领；只有附加 headless 测试验证了 `commitPlayerSelection`。

- `zone-create`

缺点：只有领域提交，不含玩家选区与地图边界渲染验证。

## 建议的补齐顺序

1. 先补一条真正的 `GameScene` 玩家验收链路。
目标：菜单选择工具 -> 指针拖拽/点击 -> 世界变化 -> 渲染反馈断言。

2. 先做最关键的 4 条 happy path。
建议顺序：`INTERACT-001 / MAP-002`、`INTERACT-002 / BUILD-001`、`INTERACT-003 / BUILD-002`、`WORK-001 / WORK-002`

3. 再补时间与 UI 表现层。
重点：`TIME-003`、`UI-002`、`UI-004`

4. 最后补异常场景。
重点：`BUILD-003`、`MAP-004`、`WORK-003`、`WORK-004`、`NEED-003`
