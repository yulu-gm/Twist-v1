# 审计报告: src/game/behavior/a-star-pathfinding.ts

## 1. 漏做需求

- **[指控]**：寻路仅依据 `WorldGridConfig` 与 `isWalkableCell`（静态 `blockedCellKeys`），算法接口未纳入「占用」维度；无法在单次查询中与占用信息联合判定可达性。
- **[依据]**：`oh-code-design/地图系统.yaml` 中「移动查询」流程写明：行为系统请求后，由「地图网格与**占用管理器**联合返回可达候选」。当前 `findPathAStar` 不接收 `OccupancyMap` 或等价快照，也未调用 `occupancy-manager` 相关能力；若调用方不把他人占用格合并进 `blockedCellKeys`，则与设计描述的联合查询不一致。（对照：`src/game/map/occupancy-manager.ts` 已存在占用模型，但本文件未与之衔接。）

- **[指控]**：地图格可变通行代价（地形/移动成本）未体现；步长恒为 `1`，启发式为曼哈顿距离，仅适合均匀四邻网格。
- **[依据]**：`oh-code-design/地图系统.yaml`「扩展点」列明：「地图格可增加地形成本、室内外标签、光照等属性」。当前实现未暴露可插拔的边代价或邻居代价，属相对设计扩展方向的预留缺口（非阻塞性故事需求，但与架构扩展表述不一致时可迭代）。

- **[指控]**：未校验起点格是否可行走；若 `start` 落在阻挡格上，仍会从该格展开搜索，可能与「可通行」语义冲突。
- **[依据]**：同文件对 `goal` 使用 `isWalkableCell` 前置校验（第 57 行），对 `start` 无对称校验；与 `oh-code-design/地图系统.yaml` 中地图格「可通行状态」作为移动基础的一般语义存在边缘不一致。

## 2. 无用兼容与 Mock

- 未发现明显问题：文件中无 `mock`、`temp`、`TODO` 或明显仅为兼容旧系统而存在的分支。

## 3. 架构违规

- **[指控]**：A* 实现放在 `src/game/behavior/` 下，与 `oh-code-design` 中对「路径相关基础查询」归属空间规则层 / 地图子系统的划分存在张力。
- **[依据]**：`oh-code-design/地图系统.yaml`「空间规则层」职责包含「提供矩形选区与路径相关基础查询」；`oh-code-design/行为系统.yaml`「风险」亦指出若行为系统同时承担路径计算等，职责会过重。本文件虽只依赖 `../map/world-grid`、未越权访问 UI 或实体状态，**从依赖方向看合理**，但**模块放置位置**更贴近地图「规则/查询」而非「决策/状态机」，属于分层边界上的偏离。

- **正向说明**：本文件仅只读使用 `WorldGridConfig` 与网格工具函数，无直接修改实体、占用或 UI 数据，未违反典型的跨层写入禁忌。

## 4. 修复建议

- **[行动点 #0022]**：为寻路增加与 `oh-code-design/地图系统.yaml`「移动查询」一致的输入契约：例如增加可选参数 `occupancy: ReadonlyMap<string, string>` 与 `selfEntityId`，在扩展邻居时将「被他人占用的格」视为不可走；或由上层构造「已合并占用」的临时 `WorldGridConfig`/阻挡集并**文档化**该契约（避免静默漏掉动态阻挡）。
- **[行动点 #0023]**：在 `findPathAStar` 开头对 `start` 做与 `goal` 对称的 `isWalkableCell` 校验；不可走则返回 `undefined`，避免从非法起点生成路径。
- **[行动点 #0024]**：将本模块迁至 `src/game/map/`（或独立的 `path-query` 小模块），由 `behavior` 中的规划器调用，以贴合「地图系统负责路径基础查询、行为系统负责决策与状态」的分层描述。
- **[行动点 #0025]**：若未来需要地形代价，将单步代价抽象为 `(grid, from, to) => number` 或在 `WorldGridConfig` 中提供格代价表，并相应将启发式改为可接纳该代价的可采纳启发式（或明确文档说明仍为均匀网格专用实现）。
