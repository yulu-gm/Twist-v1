# 审计报告: src/game/behavior/action-scorer.ts

## 1. 漏做需求

- **[指控]** 行动评分未纳入「与小人的距离」因素。`scoreActions` 仅用工单 `priority` 与需求舒适度相乘得到工作分，未结合 `WorkOrder.targetCell` 等可算距离的信息（当前 `BehaviorContext` 也未提供小人格子坐标，但评分器侧完全未体现距离维度）。
- **[依据]** `oh-gen-doc/行为系统.yaml` 中「工作决策」列明决策因素含「与小人的距离」；`oh-code-design/行为系统.yaml` 中「行动评分器」职责写明需综合「紧急度、**距离**、当前携带状态、时间语境」。

- **[指控]** 未考虑「当前携带状态」对工作选择的影响。
- **[依据]** 同上两处文档中「小人当前携带状态」/「当前携带状态」。

- **[指控]** 未使用「时间语境」。`BehaviorContext` 已包含 `time.currentPeriod` 与 `time.minuteOfDay`，但本文件未读取；无法实现设计层强调的昼夜驱动下对休息倾向、行为切换的加权（与 `oh-gen-doc/时间系统.yaml`、`oh-code-design/行为系统.yaml` 中「夜晚休息」「来自时间系统的时段事件」等叙述不对齐）。
- **[依据]** `oh-code-design/行为系统.yaml` 中行动评分器职责「时间语境」及接口边界「来自时间系统的时段事件」。

- **[指控]** 需求侧优先级未从需求子系统抽象输入，而在本模块内用饱食度/精力线性缺额自建紧急分；与设计「来自需求系统的需求优先级」接口边界及「需求系统发出紧急需求信号」流程相比，缺少统一的需求信号来源，易与阈值类规则（如别处的 `needActionSuggestion`）不一致。
- **[依据]** `oh-code-design/行为系统.yaml` 接口边界「来自需求系统的需求优先级」；关键流程「工作被需求打断」中「需求系统发出紧急需求信号」。

- **[指控]** 「行动候选」在设计中含「目标对象」等字段语义；本实现中 `eat`/`rest` 候选仅有 `kind` 与分数，无具体食物/床铺等 `targetId`，不利于后续执行层锁定目标与可解释性对齐设计数据模型。
- **[依据]** `oh-code-design/行为系统.yaml` 核心数据「行动候选」关键字段含「目标对象」。

## 2. 无用兼容与 Mock

- 本文件未发现 `mock`、`temp`、`// TODO` 或明显仅为兼容旧系统的分支，**无明显问题**。
- **补充说明**：`BehaviorContext` 中的 `time`、`behaviorState` 以及 `map.reachableCellCount` 在本评分路径中均未被读取；这不属于「多余兼容代码」，但会造成「上下文类型看似完整、评分维度实际偏窄」的落差，若长期不消费，需在上层或本模块明确是刻意延后还是遗漏。

## 3. 架构违规

- **[指控]** 行动评分器在分层上归属「决策输入层 / 行动评分器」职责，设计要求其综合多源信号；当前实现仅强依赖 `needState`、`candidateWorks` 及 `map` 的 `foodReachable`/`bedReachable` 布尔，**职责范围窄于设计描述**，属于对 `oh-code-design` 模块边界的缩水实现，而非典型的 UI 越权或跨层写状态。
- **[依据]** `oh-code-design/行为系统.yaml` 中「决策输入层」「行动评分器」职责与「接口边界」列举的输入源。

- 本文件为纯函数、无直接修改世界状态、无对 UI 或渲染的依赖，**未发现**典型的分层越权调用（如表现层改核心数据）。

## 4. 修复建议

- **[行动点 #0026]** 在决策输入层扩展 `BehaviorContext`（或并行传入评分专用 DTO）：小人当前格子、携带物/手持摘要、需求子系统给出的优先级或「是否紧迫」标志，使评分器能落地距离与携带规则，并与 `needActionSuggestion` 等阈值逻辑对齐或显式分工。
- **[行动点 #0027]** 工作分在 `priority` 与舒适度之外，增加基于 `pawnCell` 与 `WorkOrder.targetCell` 的距离衰减项（或交由工作调度器预过滤后再评分），以满足策划文档中的距离因素。
- **[行动点 #0028]** 使用 `context.time` 对 `rest`/`work`/`wander` 加权（例如夜间提高休息权重），并与时间系统事件约定一致，避免昼夜行为仅靠别处分支硬编码。
- **[行动点 #0029]** 在食物/床铺解析结果可用时，为 `eat`/`rest` 候选补充 `targetId`（或等价目标句柄），使候选列表与「行动候选」数据结构及执行协调层衔接。
- **[行动点 #0030]**（可选，对齐设计扩展点）将评分规则抽为可替换策略（性格/职业），保留当前公式为默认实现，便于验收与 A/B 规则对比。
