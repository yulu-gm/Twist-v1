# 审计报告: src/game/flows/chop-flow.ts

## 1. 漏做需求 (Missing Requirements)

- **[指控]**：`oh-gen-doc/工作系统.yaml` 中「伐木」执行流程要求小人在读条伐木前「移动到树木旁边的地图格」。当前 `runChopFlowScenario` 在认领/结算伐木工单前未将小人位置与树格对齐，仅在伐木结算之后、认领拾取工单前调用 `ensurePawnAtCell`（约第 120 行）。若真实游戏中认领伐木仍以「小人已在邻格」为前置条件，则本编排层未覆盖该步骤，与策划描述的时序不一致。
- **[依据]**：`oh-gen-doc/工作系统.yaml` 工作类型「伐木」→ 执行流程 → 步骤「小人移动到树木旁边的地图格」及后续「小人开始读条」。
- **[指控]**：`oh-code-design/实体系统.yaml`「树木转木头」流程写明伐木完成后应「同步更新地图占用与工作结果」。本文件仅通过 `settleWorkSuccess` 等间接依赖下层实现，编排层本身未体现或校验该同步是否发生；若结算层未完整落地，此处无法从代码上满足设计条款（属跨模块风险，需在结算/地图模块对账）。
- **[依据]**：`oh-code-design/实体系统.yaml` 关键流程「树木转木头」步骤列表。

（其余与设计对齐之处：`oh-code-design/工作系统.yaml`「伐木链路」中的标记、生成伐木工、完成后木头与派生拾取、在提供存储区时衔接搬运（`generateHaulWork` + 认领/结算），与实现主干一致。）

## 3. 架构违规 (Architecture Violations)

- **[指控]**：`ensureTreeMarkedForLogging`（约第 54–60 行）在编排函数内直接 `entityRegistry.replace` 写入树木 `loggingMarked`。`oh-code-design/交互系统.yaml` 将「伐木标记」归为玩家选择工具后的交互结果；`oh-gen-doc/实体系统.yaml` 亦将「标记伐木」描述为玩家框选触发。由伐木链路编排器静默补标，可能绕过交互/领域规则层对「仅已标记树可伐」的约束，使调用方在未走交互路径时仍能通过本函数伐木，与「输入来自交互系统的标记结果」的边界表述存在张力。
- **[依据]**：`oh-code-design/工作系统.yaml` 接口边界输入「来自交互系统的标记结果」；`oh-gen-doc/交互系统.yaml`「伐木标记模式」；`oh-gen-doc/实体系统.yaml`「标记伐木」触发说明。
- **[指控]**：`ensurePawnAtCell`（约第 63–68 行）在注册表中直接改写小人 `cell`，属于「瞬移」式状态修正，而非由行为/移动系统驱动。`oh-code-design/实体系统.yaml` 分层中「应用编排层」协调多实体更新、「领域规则层」负责合法性；此处若用于非测试场景，易与真实移动、占用与行为状态不一致，形成面条式状态修补。
- **[依据]**：`oh-code-design/实体系统.yaml` 分层「领域规则层」「应用编排层」职责划分。
