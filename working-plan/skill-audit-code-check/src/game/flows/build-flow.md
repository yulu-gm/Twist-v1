# 审计报告: src/game/flows/build-flow.ts

## 1. 漏做需求 (Missing Requirements)

- **[指控]**：从「玩家意图 → 蓝图」的路径上，未体现设计中的**建造校验**与**地图空间合法性**输入。编排函数在取得 `spec` 后直接调用 `createBlueprint`；而 `oh-code-design/建筑系统.yaml` 在「建筑规划层」与模块「建造校验器」中要求：校验覆盖格、检查目标格是否允许放置，并在接口边界中写明输入包含「来自地图系统的空间合法性结果」。当前 `build-flow` 未调用任何校验器门面，也未接入地图合法性结果（`createBlueprint` 本身亦未见放置冲突校验，见 `src/game/building/blueprint-manager.ts`，属整条编排链上的缺口，本文件为入口之一）。
- **[依据]**：`oh-code-design/建筑系统.yaml` — 分层「建筑规划层」、模块「建造校验器」、接口边界「来自地图系统的空间合法性结果」。
- **[指控]**：与策划文档中的**完整施工阶段**相比，本入口在单调用内顺序执行 `claimWork` 与 `settleWorkSuccess`，跳过了「小人移动至蓝图旁、读条累进进度至 100%」的游玩过程。`oh-gen-doc/工作系统.yaml` 约定建造完成条件为蓝图进度达 100%；`oh-gen-doc/行为系统.yaml` 描述建造中为读条与进度表现。若 `runBuildFlow*` 面向真实游玩主链路，则与上述阶段模型不一致；若仅作集成测试/脚本编排，则文档未在本文件声明，易产生「生产误用」风险。
- **[依据]**：`oh-gen-doc/工作系统.yaml`（建造：读条、完成条件）；`oh-gen-doc/行为系统.yaml`（建造中状态）；对照 `work-settler.ts` 注释「行为层回报成功时结算」——本流程直接结算，未经过行为层回报。
- **[指控]**：`oh-code-design/建筑系统.yaml`「关键流程 · 绘制墙体蓝图」要求笔刷路径、**批量**创建墙体蓝图与校验器筛选可放置格；本文件 `runBuildWallScenario` 仅支持单锚点单次 `createBlueprint`，未覆盖批量墙体场景（可能由上层交互另行循环调用，但本模块未体现与设计流程的对齐说明）。
- **[依据]**：`oh-code-design/建筑系统.yaml` — 关键流程「绘制墙体蓝图」步骤列表。
- **[指控]**：设计在风险中强调自动分配床铺应能记录原因以利后续手动调整；本编排仅调用 `assignBedToPawn` 并返回成功/失败类型，**未**在流程层产生或传递「归属原因」类数据（若全局无该字段则构成对设计风险项的持续暴露）。
- **[依据]**：`oh-code-design/建筑系统.yaml` — 风险「若自动分配床铺不记录原因，后续玩家手动调整会难以兼容」。

## 2. 无用兼容与 Mock (Useless Compatibility & Mocks)

- 未发现 `mock`、`temp`、`// TODO` 或明显死分支；文件注释明确墙体不做床铺分配，与意图一致。
- **[说明]**：`BuildFlowOptions.constructWorkId` 允许覆盖工单 ID，便于测试确定性；若仅测试需要，长期暴露在公共 API 上可能增加误用面，但不属于典型「无用兼容」残留，故标为低风险可选收敛项。

## 3. 架构违规 (Architecture Violations)

- **[指控]**：`oh-code-design/建筑系统.yaml` 将「建成后床铺分配」归为建筑系统能力，并由模块「归属规则器」承担「筛选尚未拥有床铺的小人」与生成归属结果；本文件在 `game/flows` 中直接依赖 `../entity/relationship-rules` 的 `assignBedToPawn`，**床铺自动分配的发起点落在实体关系模块引用上**，而非建筑子系统对外暴露的「归属规则器/建成结算」门面，与文档中的模块边界表述不一致，属于跨层耦合方向偏离设计命名与分包职责。
- **[依据]**：`oh-code-design/建筑系统.yaml` — 模块「归属规则器」「建成结算器」、关键流程「建成后床铺分配」。
- **[指控]**：同一设计在接口边界上写明建筑系统「来自工作系统的建造完成回报」；合理实现中，床铺分配可由**建成结算路径**统一触发。当前流程在 `settleWorkSuccess` 之后由 flow 再调用 `assignBedToPawn`，职责上可行，但把「建筑完成后附加规则」的一部分固定在 `flows` + `entity` 上，削弱建筑系统作为**单一聚合点**的可维护性（与「将建筑规划、工作执行、实体落地职责分清」的目标存在张力）。
- **[依据]**：`oh-code-design/建筑系统.yaml` — 目标与分层「建筑结果层」、接口边界输入「来自工作系统的建造完成回报」。

## 4. 修复建议 (Refactor Suggestions)

- **[行动点 #0087]**：在创建蓝图前接入「建造校验器」与（如适用）地图合法性查询，或在 `blueprint-manager`/建筑门面内强制校验，使 `runBuildFlow*` 的入口符合 `oh-code-design/建筑系统.yaml` 规划层与校验器职责。
- **[行动点 #0088]**：在代码或对外文档中明确 `runBuildFlowScenario` / `runBuildBedScenario` / `runBuildWallScenario` 的定位（例如：仅验收与集成测试用同步编排，非行为树驱动的主循环路径），避免与 `oh-gen-doc` 中渐进施工描述混淆；若需生产可用，应改为由行为层完成进度再调用 `settleWorkSuccess`。
- **[行动点 #0089]**：将「建成后床铺分配」收敛为建筑模块对外 API（实现设计中的归属规则器），`build-flow` 仅调用该门面，而非直接引用 `entity/relationship-rules`，以对齐 `oh-code-design/建筑系统.yaml` 模块划分。
- **[行动点 #0090]**：若产品需要支持设计风险中提到的「手动调整床铺」，补充归属结果或原因字段，并在分配路径写入；`constructWorkId` 可考虑仅对测试包导出或加 `@internal` 类约束（按项目惯例）。
