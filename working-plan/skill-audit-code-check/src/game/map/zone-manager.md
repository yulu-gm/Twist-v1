# 审计报告: src/game/map/zone-manager.ts

## 1. 漏做需求 (Missing Requirements)

- [指控]: `createZone` 未在内部执行与 `validateZoneCells` 等价的占用与非空语义绑定；仅保证去重后非空，不校验 `occupancy`。若调用方未先走校验流程，可在已被占用的格上创建区域，与策划流程中「占用冲突、区域覆盖合法性判断」的预期不一致。
- [依据]: 见 `oh-code-design/地图系统.yaml` 中 **空间规则层** 职责（「处理通行、占用冲突、区域覆盖合法性判断」）及 **关键流程 · 创建存储区**（「选区解析器校验格集合合法性」之后「区域管理器创建存储区对象」）；当前 API 将合法性完全外置，未在区域创建入口形成强制闭环。

- [指控]: 设计在 **核心数据 · 覆盖格集合** 中列出「边界范围」「连通性信息」等关键字段；本文件仅维护/传递平面 `coveredCells` 列表，未提供边界外包或连通性相关的计算、缓存或导出接口。若「区域管理器维护区域显示边界」被理解为需在地图域内产出边界数据，则此处仅有格集合，边界需完全由其他层从格列表派生（职责上可能分散，但与设计条目的字面字段不完全对齐）。
- [依据]: 见 `oh-code-design/地图系统.yaml` 中 **核心数据 · 覆盖格集合** 与 **区域管理器** 职责（「维护区域覆盖格与区域显示边界」）。

- [指控]: `oh-gen-doc/地图系统.yaml` 中存储区属性包含「存储的物资」列表；本模块创建的 `ZoneEntity` 经 `EntityRegistry` 落库时不涉及「当前区域内物资引用集合」的维护（若需求要求区域对象内嵌该列表，则不在本文件实现范围内）。
- [依据]: 见 `oh-gen-doc/地图系统.yaml` **区域系统 · 存储区 · 属性 · 存储的物资**。

## 2. 无用兼容与 Mock (Useless Compatibility & Mocks)

- 未发现明显问题（无 `mock`/`temp`/`TODO` 等残留，未见仅为兼容旧系统而存在的分支）。

## 3. 架构违规 (Architecture Violations)

- [指控]: `createZone` 与 `validateZoneCells` 拆分为两个独立入口，区域创建路径不强制经过校验，存在「绕过空间规则层」的调用契约风险；更贴近面条式编排而非设计图中连续的「校验 → 创建」流程在单一模块边界内的保证。
- [依据]: 同上，`oh-code-design/地图系统.yaml` **关键流程 · 创建存储区** 与 **空间规则层** 职责。

- 其余：`zone-manager` 通过 `EntityRegistry` 创建/删除区域实体，与 `oh-code-design/地图系统.yaml` **接口边界 · 输入** 中「来自实体系统的实体创建、删除、移动事件」及 **实体系统** 中区域实体字段（区域类型、覆盖格、名称、接受物资类型规则）整体方向一致；`getZoneAtCell` 对多块重叠区域按 `id` 字典序取最小，属于确定性策略，`oh-code-design/地图系统.yaml` **扩展点** 虽提及「优先级、过滤规则」但未规定具体规则，不构成明确违反。

## 4. 修复建议 (Refactor Suggestions)

- [行动点 #0135]: 在 `createZone` 中增加可选或必选参数传入 `occupancy`（或等价结构），在创建前调用 `validateZoneCells`（或内联相同规则），失败时返回 `ValidationResult` 或抛出明确错误，避免未校验即可注册区域。
- [行动点 #0136]: 若需对齐「覆盖格集合」的边界/连通性字段，在地图域增加纯函数（如由 `coveredCells` 计算轴对齐边界、连通分量数）并由区域创建或投影层统一调用，或在文档中明确这些字段为派生视图、非 `zone-manager` 职责以关闭缺口。
- [行动点 #0137]: 若策划要求存储区实体持有「当前存储物资」列表，在实体模型与生命周期规则中补齐，并由工作/搬运流程维护一致性；`zone-manager` 仅负责格集合与注册时保持单一职责即可。
