# 审计报告: src/game/map/occupancy-manager.ts

## 1. 漏做需求 (Missing Requirements)

- [指控]: `占用管理器` 在架构设计中需同时记录「实体占用、建筑阻挡、临时保留」，并为移动与建造提供冲突判断；本文件仅实现「格键 → 单一实体 id」的占用表，未体现建筑阻挡与临时保留两类信息，也未区分阻挡对象与可移动占用者。
- [依据]: 见 `oh-code-design/地图系统.yaml` 中 `模块` → `占用管理器` → `职责`（记录每个格上的实体占用、建筑阻挡、临时保留；为移动与建造提供冲突判断）。当前「临时保留」由 `world-grid` 的 `ReservationSnapshot` 与行为层交互点预约承担，「地形/配置阻挡」由 `WorldGridConfig.blockedCellKeys` 等承担，与本文件职责边界和设计中的「占用管理器」合一表述不一致，存在设计语义未在单模块内闭环的风险。

- [指控]: 策划层将单格描述为可包含多个实体的引用列表，而 `OccupancyMap` 为 `Map<string, string>`，每格至多一个占用人 id，无法表达同格多实体（例如与「包含实体：实体引用列表」完全对齐的数据模型）。
- [依据]: 见 `oh-gen-doc/地图系统.yaml` 中 `地图格` → `属性` → `包含实体`（类型为实体引用列表，描述为当前位于该地图格上的所有实体）。另见 `oh-code-design/地图系统.yaml` 中 `核心数据` → `地图格` → `占用对象集合`（与「单 string 占用人」相比，集合语义未在类型与 API 上显式体现）。

- [指控]: `writeEntityOccupancy` 在循环中调用 `occupy` 却丢弃 `OccupyResult`；若调用方前提不成立（存在第三方占用），冲突会被静默忽略，仅部分格写入成功，与「冲突判断」的可观测性要求不符。
- [依据]: 同 `oh-code-design/地图系统.yaml` 建筑放置判定流程中「占用管理器校验是否与阻挡对象冲突、返回可放置/不可放置或部分可放置结果」；本批量写入 API 未向上传递冲突或聚合结果，易造成状态与预期不一致且难以排查。

## 2. 无用兼容与 Mock (Useless Compatibility & Mocks)

- 未发现明显问题（无 `mock`/`temp` 占位、`// TODO` 或明显仅为兼容旧系统的分支；`ignoreOccupantId` 与 `findBlockingOccupant` 的「忽略自身」语义与移动前自检一致，属合理用途）。

## 3. 架构违规 (Architecture Violations)

- 未发现明显问题：本文件为纯函数式格子占用工具，不直接依赖 UI/Phaser，不越权修改地图网格配置；与 `coordKey`/`GridCoord` 的耦合符合地图子系统边界。

- [补充说明（非越权但需与上对账）]: 设计文档将「阻挡校验」写在占用管理器职责内，阻挡与预约实际分散在网格配置与行为层；若严格以 `oh-code-design` 模块名为准，属于跨模块拆分而非本文件内的分层越权，但会增加「谁权威」的维护成本，建议在系统级文档或 world 组合层明确单一事实来源。

## 4. 修复建议 (Refactor Suggestions)

- [行动点 #0117]: 在 world 或地图门面层明确文档化：`occupancy-manager` 仅负责实体占格索引；建筑阻挡、地形阻挡、交互点临时预约各自归属模块，或未来将三类信息收敛到与设计一致的统一占用/阻挡查询接口。
- [行动点 #0118]: 评估 gameplay 是否需要同格多实体；若需要，将 `OccupancyMap` 从单 id 演进为每格集合或分层索引（可移动实体 / 堆叠物 / 阻挡），并与 `oh-gen-doc` 的「包含实体」列表对齐。
- [行动点 #0119]: 为 `writeEntityOccupancy` 增加冲突聚合返回值（或在调试构建中断言），避免静默部分写入；与 `checkPlacement` / `findBlockingOccupant` 的用法形成一致的前置校验策略。
