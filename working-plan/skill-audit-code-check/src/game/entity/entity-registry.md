# 审计报告: src/game/entity/entity-registry.ts

## 1. 漏做需求 (Missing Requirements)

- [指控]: `EntityRegistry` 仅提供 `getByKind`、`getByCell`、`get`/`getAll`/`snapshot`，未提供按**实体间关系**筛选或导航的查询入口（例如小人携带的物资、物资的容器/存储区、床铺归属建筑等）。
- [依据]: 见 `oh-code-design/实体系统.yaml` 中模块「实体目录」职责：「提供按类型、位置、**关系**的查询入口」。

- [指控]: `getByCell` 对 `Map` 全量遍历，时间复杂度 O(n)；若项目期望由「实体目录 / 查询投影」直接支撑高频按格查询，当前实现未体现与 `oh-code-design/地图系统.yaml` 所述「按坐标索引实体、区域、蓝图、建筑」之间的明确分工（索引若在地图层，注册表侧无配套契约或注释说明）。
- [依据]: `oh-code-design/地图系统.yaml` 分层中「读取投影层」职责含「按坐标索引实体」；`oh-code-design/实体系统.yaml` 中「实体查询投影」要求为地图显示等提供稳定查询结果。

- [指控]: `create` 将草案与 `allocateId` 结果合并后直接写入 `Map`，注册表内**不做**草案字段完备性或业务合法性校验（如必填字段、关系一致性）。
- [依据]: `oh-code-design/实体系统.yaml` 中「实体原型定义」要求「定义各类型默认值与必填约束」；「领域规则层」要求「校验实体状态变化是否合法」。若上层未统一承担校验，仅靠本文件则存在守门缺口。（注：分层上校验也可能放在规则层而非注册表，此处记为与设计分工对齐风险。）

## 2. 无用兼容与 Mock (Useless Compatibility & Mocks)

- 未发现明显问题：文件中无 `mock`/`temp` 占位、`// TODO` 或明显的旧系统兼容分支。

## 3. 架构违规 (Architecture Violations)

- 未发现明显问题：本文件仅维护 `Map` 存储与坐标/类型查询，未直接依赖 UI、交互或工作编排，整体符合「实体目录」作为集中登记与查询基座的边界。

- [指控]（快照契约风险）: `toReadonlySnapshot` 对 `pawn` 分支中 `behavior`、`currentGoal` 仍直接引用原对象，仅拷贝了 `cell`；若运行时可变对象经快照暴露给 UI/规则只读消费，可能与「提供与实现无关的实体快照结构」及只读投影的稳定性预期不完全一致。
- [依据]: `oh-code-design/实体系统.yaml` 领域模型层职责：「提供与实现无关的实体快照结构」。

- [指控]（类型严谨性）: `create` 使用 `{ ...draft, id } as GameEntity` 断言，编译期不保证草案已满足各 `GameEntity` 变体的完整字段约束。
- [依据]: 与 `oh-code-design/实体系统.yaml`「实体原型定义」强调的字段约束精神存在张力（属类型/契约层面的架构债，而非运行时越权）。

## 4. 修复建议 (Refactor Suggestions)

- [行动点 #0070]: 在实体模块内补充与设计一致的**关系向查询** API（或明确文档约定：关系统一由上层服务从 `getAll`/专用索引查询，并说明为何「实体目录」不暴露关系查询），以满足或澄清 `oh-code-design/实体系统.yaml`「实体目录」条款。
- [行动点 #0071]: 与地图/世界核心约定按格查询职责——若由地图系统维护空间索引，在注册表或协作接口处用注释或类型边界标明，避免与「按坐标索引实体」重复实现或双源不一致。
- [行动点 #0072]: 在领域规则层或 `create` 入口增加草案校验策略（或工厂函数），避免非法实体进入 `Map`；与「领域规则层」校验职责对齐。
- [行动点 #0073]: 评估将 `toReadonlySnapshot` 中对 `behavior`、`currentGoal` 等嵌套状态改为不可变拷贝或冻结策略；并考虑用类型守卫/工厂替代 `as GameEntity`，收紧 `GameEntityDraft` 到 `GameEntity` 的构造路径。
