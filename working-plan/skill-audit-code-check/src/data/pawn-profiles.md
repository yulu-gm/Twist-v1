# 审计报告: src/data/pawn-profiles.ts

## 1. 漏做需求

- **结论**：对照 `oh-code-design/实体系统.yaml` 中「小人实体」所列关键字段（位置、行为状态、当前目标、携带物、饱食度、精力值、床铺归属等），以及 `oh-gen-doc/实体系统.yaml` 中小人属性定义，**均未要求**绰号、传记、备注或展示用标签等「人物风味档案」数据；`oh-gen-doc/UI系统.yaml` 中「小人状态」亦仅描述行为与进度类反馈，未规定须实现独立档案模块。
- **依据**：因此不宜认定本文件「漏实现某条明文需求」；若产品后续将「实体详情面板」定为正式范围，应在设计文档中补充小人展示字段，再由实体读模型或等价投影供给，届时再对照验收。

## 2. 无用兼容与 Mock

- **指控**：文件头与导出名明确为 Mock（`MOCK_PAWN_PROFILES`、`mockTags`、各条 `notes` 中的「mock：」前缀），整文件为静态占位数据，属于技能所述「临时/Mock 实现」范畴；在真实档案接入前会长期存在。
- **影响**：仅内置 `pawn-0`…`pawn-4` 五条；若场景或初始生成的小人数量超过五条或 `id` 命名规则变更，`pawnProfileForId` 将返回 `undefined`，UI 侧需自行兜底（与「与模拟层 id 对齐」的注释在扩展场景下易失真）。另：`src/scenes/mock-pawn-profile-data.ts` 对该文件的转发属仓库内另一处兼容层，本文件是 Mock 的**主数据源**，而非已废弃孤岛。

## 3. 架构违规

- **指控**：人物展示用文案从 `src/data` 静态表读取，而非经 `oh-code-design/实体系统.yaml` 所述「读取投影层」或 `oh-code-design/UI系统.yaml` 所述「界面状态层订阅领域系统只读数据」聚合得到，与「以读模型驱动展示」「UI 基于统一实体视图协作」的分层意图不一致；小人身份（`PawnState.id`）与风味档案（本文件）形成**双轨数据源**，扩展详情面板时容易继续堆叠静态表而非收敛到领域投影。
- **依据**：`oh-code-design/UI系统.yaml` 分层中界面状态层职责包含「订阅领域系统只读数据并转成界面态」；`oh-code-design/实体系统.yaml` 读取投影层职责为「为 UI、交互反馈、调试视图提供只读投影」。

## 4. 修复建议

- **行动点 1**：在策划/架构文档中明确「小人展示用非玩法字段」（若有）是否属于小人实体快照或独立只读扩展；避免仅依赖代码注释约定。
- **行动点 2**：接入真实档案时，将风味数据纳入实体或专用读模型，由 orchestrator/场景加载写入可观测状态，UI 只读该投影；届时删除或收缩 `MOCK_PAWN_PROFILES`，并保证 profile 键与运行时 `PawnState.id` 生成规则单一来源同步。
- **行动点 3**：若短期内保留 Mock，应用配置或生成逻辑约束「profile 表键集 ⊇ 当前场景小人 id 集」，或在 `pawnProfileForId` 调用处统一空态 UI，避免静默缺失文案。
