# 审计报告: src/data/villager-tools.ts

## 1. 漏做需求

- 对照 `oh-gen-doc/UI系统.yaml` 中「工具栏/交互组件」所列能力（如伐木工具、物资拾取标记工具、选区工具 UI、蓝图笔刷等），本文件以 `id` + 热键槽 + 文案的方式提供了更完整的快捷工具枚举，并与命令菜单、任务标记等链路对齐；策划文档未规定热键槽位数量、字母顺序或与 `toolId` 的一一映射表，**无法**据此认定本文件缺失某条**可核验**的硬性条款。
- **注释与规格表述不一致**：文件头注释写「Q–O」，实际 `VILLAGER_TOOLS` 含 10 项，最后一槽为 `zone_create`、热键 `P`，与 `VILLAGER_TOOL_KEY_CODES` 长度一致。该注释易让读者以为仅 9 槽或不含 P，**不属于**策划 YAML 中的漏做，但会造成对「展示用条目范围」的误读；若将注释视为对内规格，则应视为规格表述错误而非功能漏做。

## 2. 无用兼容与 Mock

- 未发现 `mock`、`temp`、`TODO` 或未接入新系统的死分支；`validateVillagerToolBarConfig` 为配置一致性校验，供测试与排查使用，**不属于**无用兼容。
- 本文件自身**未**承担对已废弃 `mock-villager-tools` / `villager-tool-bar-config` 的转发逻辑（转发在 `src/scenes/` 其他文件中），无额外 Mock 层。

## 3. 架构违规

- 对照 `oh-code-design/UI系统.yaml`：`工具栏模型` 职责为管理工具入口与当前选中工具；本文件仅导出静态只读配置与键盘码常量，**不**直接修改领域状态、**不**承担输入采集或模式切换，未违反「以读模型驱动展示，避免 UI 直接承担领域规则」等风险条款中的典型越权形态。
- `扩展点` 中写明「菜单树与工具栏可改为数据驱动配置」，将工具栏条目放在 `src/data/` 作为数据驱动配置与上述方向一致；从严可讨论「工具栏模型」是否应独占目录边界，但当前做法**不构成**明确的分层禁令违反。

## 4. 修复建议

- **修正文件头注释**：将「Q–O」改为与实际一致的「Q–P」或明确写「共 10 槽（Q–P）」，避免与 `VillagerTool["hotkey"]` 类型及数组长度不一致。
- **可选（跨文件）**：`command-menu.ts` 与 `villager-tools.ts` 均维护命令/工具相关文案与 id，长期可收敛为单一数据源（由菜单定义衍生热键栏，或反之），以减少标签漂移；此项超出本文件单独修改范围，仅作架构卫生建议。
- **校验保持**：继续保留 `validateVillagerToolBarConfig` 与单测，与设计文档中「显式交互模式」「可扩展键盘快捷切换」方向一致，便于新增工具时防止槽位与键码漂移。
