# 实体系统代码审计汇总报告 (Entity System Audit Summary)

## 1. 现状总结 (Current Status Summary)

实体系统整体上遵循了 `oh-code-design/实体系统.yaml` 的分层设计（分为领域模型层、领域规则层、应用编排层、读取投影层），核心模块（如注册表、生命周期规则、关系规则）边界清晰，没有严重的跨层违规调用。但在具体需求的落地、数据一致性校验以及投影层的完整性上存在一些缺口和隐患：

- **生命周期与状态流转校验不足**：在蓝图转建筑、树木转物资等关键流程中，规则层（`lifecycle-rules.ts`）缺乏对前置状态（如建造进度是否完成、树木是否有伐木标记）的严格校验，存在被非法调用导致状态错乱的风险。
- **关系一致性与归属语义缺失**：床铺分配原因（`AssignmentReason`）目前被固定写入，无法区分自动分配与手动指派；物资的预占状态（`reservedByPawnId`）与实际的携带/容器关系缺乏交叉校验；放下物资的逻辑仅支持落到地面，未实现写回目标区域（如存储区）的闭环。
- **查询与投影能力不完整**：`EntityRegistry` 缺乏按实体关系（如归属、携带、容器）查询的入口，且按坐标查询（`getByCell`）性能较低（O(n) 遍历）；`entity-projection.ts` 中的只读投影为了兼容旧数据结构，丢失了大量领域模型中已存在的关键字段（如物资的容器/占用信息、区域的规则字段、小人的行为状态等），且对异常区域数据存在静默兜底（固定在 `(0,0)`），违背了“统一实体视图”的目标。
- **类型与状态定义的偏差**：树木的伐木状态、物资的类型枚举与策划文档存在细微差异；`PawnState` 中存在新旧需求字段（废弃的 `needs` 与新的 `satiety`/`energy`）并行双写的问题，且与 `need-utils` 存在架构上的循环依赖。

## 2. 修改建议 (Modification Suggestions)

### 架构与接口演进
1. **完善实体查询能力**：在 `EntityRegistry` 或上层服务中补充按实体关系（如小人携带物、物资所在容器、床铺归属）的查询 API，以满足设计文档要求；明确按坐标查询的职责归属，若由地图系统维护空间索引，则注册表侧需做好对接，避免低效的全量遍历。
2. **修复投影层数据丢失**：补齐 `entity-projection.ts` 中 `resource`、`tree`、`zone`、`pawn` 投影缺失的关键字段，确保下游消费者能获取完整的实体快照；对于覆盖格为空的异常区域，应抛出错误或返回显式非法标记，严禁静默回退到 `(0,0)`。长远来看，应推动下游统一使用完整的 `ReadonlyEntitySnapshot`。
3. **解除循环依赖**：移除 `pawn-state.ts` 对 `need-utils` 的 re-export，将需求相关 API 统一收口到需求模块，打破 `pawn-state` 与 `need-utils` 之间的循环依赖。

### 领域规则与一致性加固
4. **收紧生命周期校验**：在 `transformBlueprintToBuilding` 中增加对蓝图建造进度（`buildState === "completed"`）的校验；在 `transformTreeToResource` 中增加对伐木标记的校验；在实体 `create` 入口增加草案字段完备性与合法性校验。
5. **补全关系一致性校验**：在 `relationship-rules.ts` 中增加对物资预占（`reservedByPawnId`）与实际携带/容器关系的交叉校验；扩展物资“放下”规则，支持落入存储区（`containerKind: "zone"` 及对应的区域校验）。
6. **严格对齐策划语义**：扩展 `AssignmentReason` 枚举以区分自动分配与手动指派；将树木的伐木状态与策划文档的三态（正常、已标记、伐木中）在类型或映射层面严格对齐；清理 `PawnState` 中废弃的 `needs` 字段，统一使用 `satiety`/`energy` 及需求阶段字段。
