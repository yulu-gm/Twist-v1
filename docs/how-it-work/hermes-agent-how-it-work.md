# Hermes Agent How It Work

这份文档不是按源码逐文件导览的方式讲 `NousResearch/hermes-agent`，而是按 `how-it-work` 的方式解释它最关键的运作主线：

- `Hermes Agent` 到底想解决什么问题
- 它为什么不只是一个“会聊天的 LLM 包装层”，而是一个带 `记忆`（跨会话保存和注入的信息）、`技能`（按需加载的专门知识文档）、`工具`（可执行动作）、`网关`（接入不同聊天平台）和 `自我改进` 闭环的 agent 系统
- 一条用户消息进入系统后，是怎样经过上下文组装、模型推理、工具调用、状态保存，再变成回复的
- `AIAgent`、`prompt builder`、`memory`、`skills`、`subagents`、`cron`、`terminal backends` 这些概念之间是什么关系
- 为什么这个仓库会把“长期运行的 agent”设计成一个带持久状态和可扩展边界的系统，而不是一次性的 prompt 调用

这份文档重点覆盖官方仓库和文档中最核心的工作机制，不逐行解释实现，不展开训练细节，也不深挖每个平台适配器的所有边角行为。

---

## Hermes Agent 到底在解决什么问题

很多“AI agent”其实只做了一件事：把用户输入丢给模型，再把模型输出还给用户。它们也许会加一点工具调用，但本质上仍然像一个短时记忆的聊天壳。

`Hermes Agent` 想解决的问题更大一些：

`怎样把大模型变成一个可以长期运行、跨会话记住事情、按需学习新技能、接入多个平台、会把工作委派出去，还能在使用过程中逐步改进自己的 agent。`

这意味着它不只关心“当前这一轮回复得像不像”，还关心：

- 这个 agent 之前和你聊过什么
- 哪些信息应该被长期记住
- 遇到新任务时能不能装载专门知识
- 在 Telegram、Discord、Slack、CLI 这些入口上能不能共用一套核心能力
- 当任务太复杂时能不能拆给子 agent
- 能不能定时自己做事，而不是只在被动收到消息时响应

所以 Hermes Agent 的核心变化不是“换了一个模型”，而是：

`把大模型放进一个有状态、有记忆、有技能、有工具和有生命周期的 agent 容器里。`

## 领域核心概念

### 1. Agent

这里的 `agent` 不是单纯的模型调用，而是一个持续存在、能保留状态、能执行动作的智能体。

它之所以重要，是因为普通聊天接口默认是“每次请求一次、返回一次”，但长期协作型系统需要一个更稳定的主体，来承接记忆、工具、身份和行为策略。

agent 这个概念要解决的问题是：

- 如何把一次次零散对话变成连续关系
- 如何让模型拥有“会话之外”的工作能力

所以在 Hermes Agent 里，真正的核心对象不是“一个 prompt”，而是：

`一个会接收消息、组装上下文、调用模型、执行工具并保存结果的运行体。`

### 2. AIAgent

`AIAgent` 是仓库里的核心 agent 运行对象，也就是主循环的承载者。

它之所以需要单独被强调，是因为很多外围能力，比如工具、技能、记忆、子代理、平台适配，最后都要汇入这个核心循环里统一调度。

它要解决的问题是：

- 收到一条消息后，系统该怎么完整处理
- 不同能力模块怎样在一次 agent 运行里协作

可以把它理解成：

`Hermes Agent 的总调度器。`

### 3. Prompt Builder

`Prompt Builder` 是负责把系统提示词、用户消息、记忆、技能内容、工具说明和其他运行上下文拼装成模型可读输入的模块。

它之所以关键，是因为大模型本身看不到数据库、文件系统、平台消息记录这些“真实系统状态”，它只能看见最终被喂进去的上下文文本或结构化内容。

Prompt Builder 要解决的问题是：

- 哪些上下文应该被注入给模型
- 这些内容以什么顺序、什么形式进入推理

所以它本质上做的是：

`把 agent 的真实运行状态翻译成模型能理解的工作台。`

### 4. Tool

`Tool` 是 agent 可以调用的具体动作，比如读文件、执行命令、联网查询、调度其他能力。

它之所以要存在，是因为只靠语言模型的内部参数，很多任务做不完。模型可以规划，但真正和外界交互，必须有可执行动作接口。

Tool 要解决的问题是：

- 如何让模型不只“说”，还能“做”
- 如何把外部世界能力暴露给 agent

所以 tool 的动机很直接：

`让 agent 从文本预测器变成可行动系统。`

### 5. Tool Registry / Tool Dispatch

`Tool Registry` 是工具注册表，`Tool Dispatch` 是工具调度过程。

它们之所以重要，是因为系统里会有很多工具，不同工具有不同参数、权限和适用场景。如果没有统一注册和分发机制，工具系统会迅速失控。

它们要解决的问题是：

- 模型提出工具调用后，系统如何找到正确工具
- 工具结果怎样返回给 agent 继续推理

所以它们的作用是：

`把大量外部能力组织成一套可发现、可调用、可回流的执行框架。`

### 6. Memory

`Memory` 是 agent 跨会话保留并重新注入的重要信息。

它之所以是 Hermes Agent 的核心卖点之一，是因为普通聊天系统每次像重新开始，而长期协作需要记住用户偏好、项目背景、固定约束和持续上下文。

官方文档里核心是两类持久记忆：

- `MEMORY.md`：更偏 agent 自己应该长期记住的全局记忆
- `USER.md`：更偏和某个用户相关的稳定偏好和个人上下文

Memory 要解决的问题是：

- 哪些信息不能只留在单轮上下文里
- 下次会话开始时，系统怎样重新获得这些长期信息

所以 memory 的本质是：

`让 agent 不必每次都从零认识你。`

### 7. State / Hermes State

`State` 是运行中的会话和系统状态；`Hermes State` 更具体地指仓库里负责持久化这些状态的机制。

它之所以要单独存在，是因为 agent 不是无状态函数。消息历史、工具执行结果、线程上下文、检索结果和任务轨迹都需要保存，不然 agent 无法稳定延续工作。

它要解决的问题是：

- 会话数据存哪里
- 历史记录怎样检索、恢复和继续使用

官方架构文档提到这一层使用 SQLite 和 FTS5，这说明它不只是简单存日志，还考虑了全文检索和历史恢复。

所以 state 的核心作用是：

`给长期运行的 agent 提供可恢复的记忆底座。`

### 8. Skill

`Skill` 是按需加载的专门知识文档或工作流说明。

它之所以重要，是因为一个长期 agent 不可能把所有领域知识和所有操作规范都始终塞在主 prompt 里。那样既浪费上下文，也会让模型噪声过大。

Skill 要解决的问题是：

- 当任务涉及某个特定领域时，怎样快速注入专业做法
- 如何把复用性高的操作知识沉淀成可再调用模块

所以 skill 更像：

`给 agent 临时安装的工作手册。`

### 9. Gateway

`Gateway` 是把不同外部消息平台接到同一个 agent 内核上的入口层。

它之所以重要，是因为如果每个平台都各自维护一套 agent 逻辑，系统会很快分裂。Hermes Agent 想做的是“多个入口，共用一个大脑”。

Gateway 要解决的问题是：

- Telegram、Discord、Slack、CLI 等消息怎样统一进入核心 agent
- 平台差异怎样在边界层被吸收

所以 gateway 的设计动机是：

`把多平台接入问题和 agent 核心逻辑解耦。`

### 10. Terminal Backends

`Terminal Backends` 是和不同终端或命令执行环境打交道的后端接口。

它之所以会出现，是因为 agent 如果要执行命令、读写项目、操作环境，就需要一个稳定的执行边界，而不是把“运行 shell”硬编码在核心逻辑里。

它要解决的问题是：

- 命令在哪执行
- 不同环境如何统一成 agent 可调用的执行接口

所以 terminal backend 的作用是：

`把执行环境抽象出来，让 agent 能稳定调用外部操作能力。`

### 11. Subagent / Delegation

`Subagent` 是被主 agent 派出去处理某个子任务的代理；`Delegation` 是委派机制。

它之所以重要，是因为很多复杂任务不适合所有事情都在一个上下文里完成。拆分子任务后，可以并行、隔离上下文，或让某个代理专注处理特定问题。

它要解决的问题是：

- 如何把大任务拆小
- 如何避免一个主上下文塞下所有中间过程

所以 delegation 的核心价值是：

`把单线程思考变成多工作单元协作。`

### 12. Cron

`Cron` 是定时触发任务的机制。

它之所以重要，是因为一个长期运行 agent 不应该只会“收到消息再反应”。很多真实系统需要周期性检查、提醒、汇总或执行预先安排的工作。

Cron 要解决的问题是：

- 如何让 agent 在没有人实时发消息时也能行动
- 如何支持周期性自动任务

所以 cron 的意义是：

`让 agent 从被动响应者变成可定时行动的系统。`

### 13. Self-Improvement

`Self-Improvement` 指 agent 在使用过程中把经验沉淀进记忆、技能或其他结构，从而逐步提高后续表现的能力。

它之所以被 Hermes Agent 强调，是因为很多 agent 每轮结束后什么都没留下，下一次又重新犯同样错误。Hermes Agent 更像是在尝试做一个学习闭环。

它要解决的问题是：

- 怎样让一次任务的经验影响下一次任务
- 怎样避免 agent 永远停留在“短时聪明、长期失忆”

所以自我改进不是神秘能力，而更接近：

`把一次次使用结果沉淀成可复用结构。`

## 这些概念之间的关系

可以把 Hermes Agent 的工作主线理解成下面这条链：

1. 用户从 `Gateway` 或 CLI 等入口发来一条消息
2. 消息进入核心 `AIAgent`
3. `Hermes State` 负责提供已有会话和持久状态
4. `Memory` 把长期信息取出来，准备注入
5. 如果当前任务命中了某些 `Skill`，对应知识文档会被按需加载
6. `Prompt Builder` 把系统规则、消息历史、记忆、技能、工具说明等拼成模型上下文
7. 模型开始推理
8. 如果模型判断需要行动，就通过 `Tool Dispatch` 调用某个 `Tool`
9. 工具执行可能会走到 `Terminal Backends`、检索层、外部服务或其他系统边界
10. 工具结果再回流给 `AIAgent` 继续推理
11. 如果任务很复杂，主 agent 还可以委派给 `Subagent`
12. 最终形成回复，返回给原消息入口
13. 处理过程中产生的重要信息再写回 `State`、`Memory`、甚至新的 `Skill`
14. 如果某些任务要长期定期执行，则由 `Cron` 在未来触发

所以 Hermes Agent 不是“模型 + 一点工具”，而更像：

`一个以 AIAgent 为核心、以 prompt 组装为输入面、以工具和终端为行动面、以 state 和 memory 为长期持续面、以 skills 和 subagents 为扩展面的 agent 操作系统。`

## 业务场景解释

### 场景一：用户在 CLI 里发一条消息后，内部到底发生了什么

假设用户在命令行里对 Hermes Agent 说：

`帮我总结这个仓库的架构，并记住我更喜欢中文输出。`

内部更接近下面这个过程：

1. CLI 把消息送进核心 agent
2. agent 从持久状态中恢复当前会话历史
3. 系统读取已有长期记忆，看是否已经知道这个用户的语言偏好
4. 如果还不知道，这次任务里这个偏好就有可能在后续被写进 `USER.md`
5. Prompt Builder 把系统说明、消息历史、已有记忆、工具说明等拼成模型输入
6. 模型先判断是否需要工具，比如读文件、搜索代码、查看目录
7. 如果需要，工具系统执行相应动作并返回结果
8. 模型结合工具结果组织回答
9. 最终回复返回给 CLI
10. “更喜欢中文输出”这种稳定偏好，后续可能被沉淀进用户记忆

这个场景说明：

- 回复不是直接由当前消息单独决定的
- 记忆和工具都参与了这一轮工作
- 一次会话的结果可能改变未来的会话行为

### 场景二：为什么它能在 Telegram、Discord、Slack 上共用一套 agent 内核

如果没有 gateway，每个平台都得自己处理：

- 消息接收
- 用户身份映射
- 会话线程
- 回复发送
- 工具权限和行为边界

这样系统会迅速复制粘贴成很多份。

Hermes Agent 的设计更接近这样：

1. 不同平台适配器负责接收各自格式的消息
2. 它们把消息统一转换成核心 agent 可理解的内部事件
3. 核心 agent 用相同的方式处理上下文、记忆、技能、工具和推理
4. 最终再把输出交回平台层，由各平台负责发送

这个场景说明 gateway 的本质不是“多写几个机器人接口”，而是：

`让平台差异留在边缘，让 agent 智能留在核心。`

### 场景三：跨会话记忆为什么是 Hermes Agent 的关键能力

很多聊天系统在这一轮会说：

`好的，我记住了你喜欢中文。`

但下次新会话开始，它其实已经忘了。

Hermes Agent 试图解决的就是这个落差。更接近的机制是：

1. 系统判断这条信息是否值得长期保存
2. 如果值得，它会写入某种持久记忆结构，比如 `USER.md` 或 `MEMORY.md`
3. 下次会话开始时，这些信息会被重新读取
4. Prompt Builder 再把这些内容注入模型上下文
5. 模型因此从一开始就带着这些背景来回答

这个场景说明记忆的关键不是“存起来”本身，而是：

`在后续会话开始时，重新进入模型可见上下文。`

否则它只是数据库里的一条死记录。

### 场景四：技能系统为什么比把所有说明永久塞进主 prompt 更合理

假设 agent 要帮你做一件专业任务，比如分析某个特定代码库、遵循某家公司内部流程，或按照某种固定格式生成报告。

最笨的方式是把所有知识永远塞进主系统提示词。但这样会有几个问题：

- 上下文过长
- 很多任务根本用不到这些说明
- 无关内容会污染推理

Hermes Agent 的技能系统更接近：

1. 平时只保留较通用的核心规则
2. 当任务命中特定领域或流程时，再按需加载那个 skill
3. skill 内容作为临时但高权重的操作手册注入当前上下文
4. 任务完成后，主系统不需要长期背着所有细节

这个场景说明 skill 的核心价值是：

`让 agent 的专业知识模块化、可装载，而不是永久堆在主脑里。`

### 场景五：为什么 Hermes Agent 需要子代理委派

假设一个复杂任务同时包含：

- 阅读大量代码
- 搜索外部资料
- 生成对比分析
- 整理最终结论

如果所有步骤都塞给一个 agent 在同一上下文里处理，很容易出现：

- 上下文过载
- 中间探索过程污染最终回答
- 主 agent 被细节拖住

子代理机制提供了另一种做法：

1. 主 agent 保留总体目标和最终整合责任
2. 某个探索型子任务被派给 subagent
3. 子代理在自己的上下文里运行、查找、执行
4. 它返回压缩后的结论给主 agent
5. 主 agent 再把多个子结果整合成最终输出

这个场景说明 delegation 的意义不是炫技，而是：

`通过上下文隔离和任务拆分，提高复杂任务的稳定性。`

### 场景六：Cron 为什么让它更像长期 agent

再看定时任务。比如你希望 agent：

- 每天早上汇总某个频道的新消息
- 每周提醒你跟进某个项目
- 周期性检查某些状态并生成报告

如果没有 cron，这类工作必须每次由人手动触发。

有了 cron 后，系统可以：

1. 保存未来要执行的任务定义
2. 在对应时间自动唤起 agent 运行
3. 让 agent 像处理普通消息一样处理这次任务
4. 把结果发送到指定渠道或保存进系统状态

这个场景说明 cron 的本质是：

`把 agent 从“被消息唤醒”扩展成“也能按时间自主行动”。`

## 这套机制里最该记住的规则

### 1. Hermes Agent 的核心不是聊天，而是长期运行的 agent 生命周期

消息、记忆、技能、工具、状态和调度都围绕这个生命周期组织。

### 2. AIAgent 是核心调度中枢

外部入口再多，最后都要汇入这个核心循环，由它统一处理上下文、推理和行动。

### 3. Prompt Builder 决定了模型到底“看见了什么”

记忆、技能、工具说明和会话历史，只有被正确注入，才会真正影响模型行为。

### 4. Memory 的关键不是存，而是“下次还能重新进入上下文”

这才是跨会话记忆真正起作用的地方。

### 5. Skills 解决的是专业知识和流程的模块化装载

它避免把所有知识永久塞进主 prompt，降低噪声并提升复用性。

### 6. Tools 和 Terminal Backends 让 agent 真正能行动

没有它们，模型再会规划，也只是停留在文字层面。

### 7. Gateway 让多平台共享同一个大脑

它把平台差异压在边界层，避免核心 agent 逻辑被分裂。

### 8. Subagents 和 Cron 说明这个系统不是一次性助手，而是可拆分、可调度、可持续运行的 agent 平台

这也是它和很多“单轮聊天代理”最大的区别之一。

## 可以怎么理解整个系统

如果把 Hermes Agent 压缩成一句话，可以这样记：

`Hermes Agent 是一个把大模型放进长期运行容器里的系统：消息从多平台入口进入，核心 AIAgent 结合状态、记忆、技能和工具组装上下文并做推理，再通过工具、子代理和定时机制与外界持续交互。`

如果再压缩一点，可以记成：

`它不是“会聊天的模型壳”，而是“有记忆、有技能、有工具、有状态、有调度的长期 agent 内核”。`
