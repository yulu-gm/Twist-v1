### how-it-work是什么

how-it-work文档的目的：让几乎不懂某个专业领域的读者能用比较快的时间对该领域建立心智模型，并理解该领域60~80%的核心知识。

以下是为一个领域撰写一篇 `how-it-work` 文档的格式

（注意，下面附带的例子都只是简单举例，你输出时需要更加详细和易懂）

**领域核心概念**

1. 先列出场景中高频出现、处于核心地位的概念
   例如：Web 请求处理中的 `request`、`response`、`handler`
2. 然后解释这些概念是什么，这些核心概念之间以及与整个领域内外的关系
   例如：`request` 是客户端发给服务端的一次请求，`handler` 是处理该请求的代码入口，`response` 是服务端返回给客户端的结果
3. 专业名词首次出现时，如果当前上下文没有立刻解释它，则补一个简短注释

   判断是否要注释的标准：必须把读者视作几乎完全不懂该领域的知识水平，任何在本领域有专有含义的词都需要解释。
   例如：`ETF`（可像股票一样买卖的基金）、`滑点`（预期成交价和实际成交价的差）
4. 解释核心概念或者专业名词时，除了解释这个东西是什么，还必须介绍这一事物出现的原因或动机
   例如：高级编程语言（如C语言等比汇编语言封装度更高的语言。用来解决汇编语言封装度太低的问题）
   例如：越位（指进攻方传球时，接球者位置不得超过第二名防守球员，这是为了避免进攻方球员长期蛰伏在防线后方导致场面不好看而设计的规则）

**业务场景解释**

1. 用业务场景解释在真实业务中，这一整个领域是如何工作的
   例如：用户在前端点击“提交订单”按钮后，浏览器会发出一个 `request`，服务端会根据 URL 将请求交给对应的 `handler` 处理，`handler` 在完成校验和数据操作后返回 `response`，浏览器再根据 `response` 更新页面状态
