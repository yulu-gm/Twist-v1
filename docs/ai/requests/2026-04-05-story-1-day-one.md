# 主控需求单：环世界第一天（story-1）

来源：`oh-story/story-1.md`  
路由：按 `.agent/skills/route-demand` 提炼与拆分（本轮仅产出主控需求单与集成说明，各系统 aidoc 待分派补写）。

## 假设与缺口（需在实现前收敛）

- 「白天」若需与模拟时间挂钩：当前注册表无独立 `time` 系统，可以先由 `scene-hud` 展示固定或者写死的时段，或者在未来拆分出世界时间之后再接入。
- 「存储区、可拾取标记、物资堆叠与归属、伐木读条、树转换为木头」涉及权威世界状态：现有索引以 `world-grid` 、 `task-planning` 、 `pawn-state` 为主；如果掉入物品与工作队列需要进入 `src/game/`，应当在对应系统的 aidoc 中明确从场景 mock 迁移到领域内的顺序。
- 下文「状态承诺」在领域未确定处使用 **承诺意图** 描述，接口细节保留在集成文档与后续的 aidoc 中。

## 场景需求条目（按 story 顺序）

每条结构：玩家目标 → 输入动作 → 画面反馈 → 状态承诺 → 边界条件。

### R-01 开局：小人与白天

| 维度 | 内容 |
|------|------|
| 玩家目标 | 进入关卡后立即看到小人在地图上，且处于白天。 |
| 输入动作 | 无（进入场景即生效）。 |
| 画面反馈 | 地图可见；至少一名小人可见；环境光或 UI 提示为白天（与项目美术一致即可）。 |
| 状态承诺 | 小人出生/占位在合法格上；初始时段为白天（或等价展示状态）。 |
| 边界条件 | 时间可为 HUD 假数据；出生点可写死列表，但须与 `world-grid` 可走规则一致。 |

### R-02 初始散落物资

| 维度 | 内容 |
|------|------|
| 玩家目标 | 地图上散落可供后续操作的初始物资。 |
| 输入动作 | 无（生成世界时布置）。 |
| 画面反馈 | 物资在格子上可辨认（图标/色块/占位均可）。 |
| 状态承诺 | 每条物资关联格子坐标与类型（原型可用 mock 列表）；不与非法占用规则冲突（或明确允许重叠的临时规则）。 |
| 边界条件 | 用户界面优先可以先只做 `scene-hud` 或 `mock-ground-items` 同类方案；进入模拟层需要 `route-demand` 补充 aidoc。 |

### R-03 无工作时闲逛

| 维度 | 内容 |
|------|------|
| 玩家目标 | 没有可执行工作时，小人看起来在「漫无目的」移动。 |
| 输入动作 | 无。 |
| 画面反馈 | 小人持续在可走区域移动（非静止摆拍）。 |
| 状态承诺 | `task-planning` 在无高优工作时选用闲逛/游荡计划；移动符合 `world-grid` 阻挡规则。 |
| 边界条件 | 可以与现有 `wander-planning` 行为对齐；路径寻路精度可以按照现有测试承诺。 |

### R-04 新建存储区（菜单 + 框选）

| 维度 | 内容 |
|------|------|
| 玩家目标 | 划定一片连续（或规则允许的）格子作为存储区。 |
| 输入动作 | 选择「区域 → 存储区 → 新建」；用选区/框选工具在地图上拖选若干格。 |
| 画面反馈 | 菜单或工具条状态切换为存储区新建模式；框选过程有矩形/格子高亮；确认后存储区边界或填充可视化。 |
| 状态承诺 | 世界或会话状态中登记一块「存储区」及其占格集合；与障碍/占用规则的关系在 aidoc 中写清（允许/禁止覆盖）。 |
| 边界条件 | 工具入口与快捷键可走 `scene-hud`；框选与焦点属 `selection-ui`；格集合属 `world-grid` 或下游领域模型。 |

### R-05 标记散落物资为可拾取

| 维度 | 内容 |
|------|------|
| 玩家目标 | 指定哪些散落物应被小人搬走。 |
| 输入动作 | 框选地图上若干格或若干物资实体，执行「标记可拾取」类命令（具体入口可对标现有工具栏）。 |
| 画面反馈 | 被选目标有高亮或图标状态变化；标记后与未标记可区分。 |
| 状态承诺 | 被标记实体进入「待搬运」队列或带 `haul` 意图，供 `task-planning` 生成搬运工单。 |
| 边界条件 | 可以先通过命令转换为 mock 列表；领域层需要定义「谁接受标记、如何与存储区关联」。 |

### R-06 有工时优先收集

| 维度 | 内容 |
|------|------|
| 玩家目标 | 一旦有可执行的收集工作，小人应去执行而非继续纯闲逛（在优先级规则下）。 |
| 输入动作 | 无（依赖 R-05 已产生工作）。 |
| 画面反馈 | 小人走向物资并执行搬运相关动画或占位表现。 |
| 状态承诺 | `task-planning` 将搬运工单的优先级置于闲逛之上（具体比较规则可测）；工单关联目标物资与目标存储区。 |
| 边界条件 | 并发多个小人时的分派与拾取顺序可以简化，但是必须在 aidoc 写明确定性策略或者允许随机档位。 |

### R-07 物资进入存储区

| 维度 | 内容 |
|------|------|
| 玩家目标 | 收集结果体现在存储区内（逻辑或视觉上「到了指定区域」）。 |
| 输入动作 | 无（工步完成由模拟推进）。 |
| 画面反馈 | 物资自地面消失或转入存储展示；存储区内堆叠/列表更新（可用最小表现）。 |
| 状态承诺 | 物资位置/归属更新为「在某存储区内」或与存储区绑定的容器槽位；地面引用移除或标记已清空。 |
| 边界条件 | 存储区容量、堆叠规则未定时可 stub；禁止出现「工单完成但状态未变」的不可逆幻觉，或须在限制中写明。 |

### R-08 进入伐木流程（工具选择与框选树）

| 维度 | 内容 |
|------|------|
| 玩家目标 | 指定一片区域的树为砍伐对象。 |
| 输入动作 | 选择伐木工具；在地图上框选包含多棵树的区域（或逐棵选择，以设计为准；story 为区域框选）。 |
| 画面反馈 | 工具模式切换；选中树有高亮/标记图标。 |
| 状态承诺 | 被标记树携带「待伐木」或可派发工单类型，供 `task-planning` 消费。 |
| 边界条件 | 树若仅为视觉层对象，须定义与格的对应关系以便寻路与工位。 |

### R-09 小人执行伐木（邻位 + 读条）

| 维度 | 内容 |
|------|------|
| 玩家目标 | 小人走到树旁，读条后完成一次砍伐。 |
| 输入动作 | 无。 |
| 画面反馈 | 小人在树邻格（或约定工位）停驻；进度条或周期动画表示读条；结束后树移除或变为树桩，并出现木头物资。 |
| 状态承诺 | 工单占用树/防并发砍伐规则；读条结束时触发：树状态变迁 + 生成木头物资实体 + 释放/转工单。 |
| 边界条件 | 读条时长可配置；被打断规则可在二期单列需求。 |

### R-10 伐木产物收入存储区

| 维度 | 内容 |
|------|------|
| 玩家目标 | 与 R-06/R-07 一致：木头应被搬到既定存储区。 |
| 输入动作 | 可选：自动沿用玩家主存储区或最近存储区；若需玩家再标记，则增加与子流程。 |
| 画面反馈 | 与 R-06/R-07 同类的搬运与落地表现。 |
| 状态承诺 | 木头与地面、Pawn 携带、存储区之间的状态迁移与 R-07 一致。 |
| 边界条件 | 若未定义默认目标存储区，必须在需求或 aidoc 中写明回退策略（停留原地/报错/需玩家指令）。 |

## 系统拆分汇总（目标系统 → 玩家可见结果 → 测试层级 → 主要依赖）

| 目标系统 | 本故事簇负责的玩家可见结果 | 默认 failing test | 主要依赖 |
|----------|---------------------------|-------------------|----------|
| world-grid | 出生点、可走/阻挡、选区与存储区/标记所占格子一致、树与工位格关系 | domain | pawn-state, task-planning |
| selection-ui | 框选、目标高亮、焦点切换（存储区新建、物资标记、伐木选树） | acceptance | scene-hud, pawn-state |
| scene-hud | 「存储区新建」「标记可拾取」「伐木」等入口与模式反馈；可选白天展示 | component | selection-ui, pawn-state |
| pawn-state | 小人位置、移动/读条/携带等可供场景读取的状态（与现有模型对齐处沿用） | domain | task-planning |
| task-planning | 闲逛 vs 搬运 vs 伐木工单的选择、派发与状态推进 | domain | pawn-state |

**共享 fake/stub 清单（显式登记，避免静默混写）**

- 白天展示、部分物资列表：可先 HUD/mock。
- 存储区容量、多名工人争抢同一棵树：可以简化规则，但是必须在 aidoc 写明。
- 默认目标存储区：未选时须有定义，避免无法结算。

## 建议 TDD / 集成顺序（概要）

1. `world-grid`：选区格子集合、存储区占用格子、与可走或者阻挡一致性（domain）。  
2. `task-planning`：无工作闲逛与有搬运或者伐木工单的优先级与状态机（domain）。  
3. `pawn-state` 与场景：读条与携带表现与状态同步（domain 与必要的 component）。  
4. `selection-ui` 与 `scene-hud`：工具链路与框选验收（acceptance 或者 component）。

更细的跨界数据流见：`docs/ai/integration/2026-04-05-story-1-day-one.md`。
