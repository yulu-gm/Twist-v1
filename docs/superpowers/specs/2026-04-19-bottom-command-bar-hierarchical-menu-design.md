# 底部命令栏分层菜单设计

## 背景

当前底部命令栏本质上是一个扁平的调试式工具栏，`Build` 和 `Zone` 通过两个特判弹出菜单实现。这个结构无法自然扩展到更深层级的菜单，同时每类子菜单还在使用不同的 UI 模式，而不是统一的导航模型。

这次重构将把现有的“工具栏 + 下拉菜单”改为一个锚定在左下角的统一分层命令列表。任意层级都使用同一套正方形入口列表样式。进入子菜单时，当前列表会整体替换为该分支的子项；返回上一级时，列表恢复为父层级的内容。

## 目标

- 支持多级命令菜单，并且所有层级共用同一套交互模型。
- 将当前偏调试按钮风格的底栏改为更像游戏命令盘的正方形入口列表。
- 在不同层级之间切换时保持视觉结构稳定。
- 允许叶子命令被连续使用，而不自动关闭或重置菜单。
- 让鼠标点击与键盘快捷键共享同一份导航状态。

## 非目标

- 本次不接入图标资源，入口内部先使用文字占位。
- 本次不改具体游戏命令语义，只改命令栏导航与快捷键路由。
- 本次不重做顶部状态栏、Inspector、Toast 或其他 UI 面板。
- 本次不支持分页、多行布局或溢出滚动。

## 信息架构

命令栏任意时刻只渲染一个菜单层级的入口集合。

### 一级菜单

一级菜单固定包含以下五个入口，顺序如下：

1. `选择`
2. `建造`
3. `指令`
4. `区域`
5. `取消`

### 子菜单结构

第一版子菜单树如下：

- `建造`
  - `结构`
    - `墙`
  - `家具`
    - `床`
- `指令`
  - `采矿`
  - `收获`
  - `砍伐`
- `区域`
  - `存储区`
  - `种植区`

### 返回项规则

- 任何非根层级都会自动在第一个位置插入 `返回`。
- `返回` 不是 schema 里的真实节点。
- 只要当前菜单路径不在根层，命令栏渲染器就自动生成 `返回` 入口。

## 交互模型

### 列表整体替换

- 命令栏始终显示一排水平正方形入口。
- 点击分支节点后，整排列表替换为该分支的子节点。
- 点击 `返回` 后，整排列表替换为父层级的入口。
- 所有层级共用同一个组件和同一套视觉样式。

### 叶子项激活

- 点击叶子项时，触发现有工具激活逻辑。
- 激活叶子项后，菜单不关闭。
- 激活叶子项后，不回退到上一级。
- 激活叶子项后，当前菜单路径保持不变。

这个行为是必须的，因为玩家需要连续下达同类命令，而不希望每次都重新进入相同分支。

### Esc 行为

- 如果当前层级不是根层，`Esc` 返回上一级。
- 如果当前层级已经是根层，`Esc` 切回 `选择` 工具。
- 根层按 `Esc` 时，还需要把一级高亮重置到 `选择`。

### 外部点击行为

- 点击命令栏外部区域时，不关闭也不重置命令栏。
- 命令栏表达的是持续性的工具/导航状态，而不是临时弹窗。

## 键盘快捷键模型

当前固定工具快捷键 `Q/B/M/H/X/Z/C` 将不再用于命令栏导航。

当前可见层级中的可操作入口，将按照从左到右的顺序动态映射到以下按键：

- `Z`
- `X`
- `C`
- `V`
- `B`
- `N`
- `M`

规则如下：

- `返回` 永远使用 `Esc`，不占用上面这组字母位。
- `返回` 方块上必须明确显示 `Esc` 作为快捷键提示。
- 其他非返回方块必须显示自己当前分配到的快捷键提示。
- 只要菜单路径发生变化，快捷键映射就重新按当前可见层级计算。

### 容量约束

- 本次设计中，每个层级最多只能包含 7 个非返回可操作项。
- 本次不设计分页、滚动、溢出处理或第二排入口。
- 菜单内容必须控制在这个上限内。

## 视觉设计

### 布局

- 命令栏锚定在左下角。
- 不再横向铺满整个屏幕底边。
- 容器宽度由当前可见入口数量决定。
- 所有入口只渲染在单行内。

### 入口形态

- 每个入口都是正方形按钮。
- 当前版本使用文字作为视觉占位。
- 每个方块内部包含：
  - 主标签文字
  - 可见的快捷键提示
  - 激活态样式

### 风格方向

命令栏必须摆脱“调试控件”的观感，转成更接近游戏命令盘的视觉语义：

- 有明确的底板容器
- 有稳定的正方形轮廓
- 有清晰的边框、底色与层次关系
- 有明确的 hover / press 状态
- 有持续存在的激活态高亮

所有层级必须使用完全一致的列表外观，不允许再出现 dropdown 专用皮肤，也不允许出现独立的子菜单外壳。

## 状态归属

当前命令栏所处层级不能只放在 Preact 组件本地状态中。

菜单路径应该存入 `PresentationState`，这样 UI 层和 Phaser 键盘输入层才能操作同一份导航状态，避免键盘已经改变菜单层级但 UI 不知道，或者 UI 已经切层但输入层仍停留在旧状态。

### 建议的状态结构

在 `PresentationState` 中增加字段：

```ts
commandMenuPath: string[];
```

语义如下：

- `[]` 表示根层
- `['build']` 表示当前正在查看 `建造` 的下一层
- `['build', 'furniture']` 表示当前正在查看 `建造 -> 家具` 的下一层

具体字符串 id 应与菜单 schema 中定义的节点 id 保持一致。

## 菜单 Schema

当前的扁平 `toolActions` 数组不足以表达分层导航，不适合作为命令栏结构的唯一真实来源，应被树形 schema 替代。

建议引入如下结构：

```ts
interface CommandMenuNode {
  id: string;
  label: string;
  kind: 'branch' | 'leaf';
  action?: ToolActionDef;
  children?: CommandMenuNode[];
}
```

设计规则如下：

- 一级和子级都使用同一种节点类型。
- 只有叶子节点携带可执行的工具激活 payload。
- 分支节点只负责定义结构与子关系。
- `返回` 不属于 schema，而是在渲染阶段自动生成。

`ToolActionDef` 可以继续作为叶子节点的执行 payload 类型保留，但不应再承担整个菜单层级结构的定义职责。

## 高亮规则

高亮逻辑在不同层级之间必须保持稳定且可预测。

### 当前层级内

- 如果当前可见层级中包含已激活叶子项，则高亮该叶子项。
- 如果当前可见层级展示的是某个父层，而激活项位于其子树中，则在父层中高亮对应分支节点。

### 根层高亮

在根层时：

- 当前激活工具是 `选择`，则高亮 `选择`
- 当前激活叶子属于 `建造` 子树，则高亮 `建造`
- 当前激活叶子属于 `指令` 子树，则高亮 `指令`
- 当前激活叶子属于 `区域` 子树，则高亮 `区域`
- 当前激活工具是 `取消`，则高亮 `取消`

示例：

- 当前激活 `墙`，根层高亮 `建造`
- 当前激活 `采矿`，根层高亮 `指令`
- 当前激活 `存储区`，根层高亮 `区域`

## 键盘输入集成

当前键盘系统在 `src/adapter/input/keyboard-bindings.ts` 中直接处理 `Esc` 和各类工具热键。重构后：

- `Esc` 必须优先作用于 `commandMenuPath`
- `Z/X/C/V/B/N/M` 必须触发当前可见菜单项，而不是固定全局工具
- 原有固定工具热键 `Q/B/M/H/X/Z/C` 需要从这套命令栏流程中移除

所有通过键盘触发的菜单导航，都必须更新同一个 `PresentationState.commandMenuPath`，与 UI 层保持同步。

## 实现边界

预期涉及的代码区域如下：

- `src/ui/domains/build/components/tool-mode-bar.tsx`
  - 去掉现有本地 submenu 布尔状态，改为根据共享菜单树和 `commandMenuPath` 渲染
- `src/ui/domains/build/build.schemas.ts`
  - 用分层菜单 schema 替换或扩展现有扁平工具栏 schema
- `src/ui/domains/build/build.selectors.ts`
  - 增加当前可见菜单层级、父级/叶子高亮状态、快捷键提示等 selector
- `src/presentation/presentation-state.ts`
  - 增加 `commandMenuPath` 以及对应的 push / pop / reset 辅助逻辑
- `src/adapter/input/keyboard-bindings.ts`
  - 将 `Esc` 和 `Z/X/C/V/B/N/M` 路由到菜单导航，而不是固定工具热键
- `src/ui/styles/app.css`
  - 用左下角方块命令盘样式替换现有全宽工具栏与 dropdown 样式

## 测试策略

应在三个层面补充测试。

### Selector / State 测试

- 根层能解析出五个一级入口
- 进入 `建造` 后能解析出 `返回 / 结构 / 家具`
- 进入 `建造 -> 家具` 后能解析出 `返回 / 床`
- 当当前激活叶子位于子树中时，根层高亮能正确落在父分支
- 根层 `Esc` 行为会把激活工具重置为 `选择`
- 非根层 `Esc` 会把 `commandMenuPath` 正确弹出一级

### 组件测试

- 命令栏能根据当前层级渲染正确的方块入口
- 点击分支节点会整体刷新列表为子项
- 点击 `返回` 会恢复父层列表
- 点击叶子项只触发激活，不改变当前层级
- 每个可见方块都会显示正确的快捷键文字

### 键盘测试

- `Z/X/C/V/B/N/M` 触发的是当前可见入口，而不是固定工具
- 非根层按 `Esc` 会退一级
- 根层按 `Esc` 会切回 `选择`

## 风险与防护

### 风险：UI 与输入层漂移

如果菜单导航状态只存在于组件本地，Phaser 键盘输入和 Preact 渲染很容易出现状态漂移。将菜单路径存入 `PresentationState` 是这个风险的核心防护。

### 风险：单组件继续膨胀

`tool-mode-bar.tsx` 现在已经有 `build/zone` 两套特判逻辑。此次重构应删除这类特判，而不是继续在它上面叠更多条件分支。

### 风险：未来菜单扩展

当前菜单树符合“单层最多 7 项”的上限，但如果后续某个层级继续变宽，就需要后续单独设计分页或溢出方案。这个问题明确不在本次范围内。

## 验收标准

- 底部命令栏变为左下角方块入口列表，而不是全宽工具栏
- 根层和子菜单共用同一套分层列表模型
- 进入子菜单后，当前列表整体替换为子项列表
- 所有非根层都会自动补出 `返回`，并支持 `Esc` 返回
- 点击叶子项后，当前菜单层级保持不变
- 根层按 `Esc` 会切回 `选择` 并高亮 `选择`
- 当前可见入口会动态显示 `Z/X/C/V/B/N/M` 快捷键提示，`Esc` 专用于 `返回`
- `Build` 与 `Zone` 不再依赖各自独立的下拉菜单实现
