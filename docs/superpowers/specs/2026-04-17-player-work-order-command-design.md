# 玩家订单化指令与全局优先级看板设计

**日期：** 2026-04-17  
**主题：** 将玩家下达工作的主要交互改造为“订单化指令系统”：地图操作或非地图结果请求都会生成需求单，殖民者只从全局订单列表中取活，玩家通过统一看板查看进度、排序优先级、暂停和取消。

---

## 1. 背景

项目方向已经从“较强微操的 RimWorld 式 colony sim”转向更强调高层经营、扩张与生产组织的玩法。当前代码中，地图框选与 `designation` 更接近“直接给地图打可执行标记”，而 AI 侧会直接从世界中的工作对象寻找活干。这种模式的问题是：

- 玩家真正控制的对象不是统一的“需求”，而是分散在地图上的裸标记。
- 小人直接从地图目标取活，玩家难以从一个入口看到“现在 colony 正在做哪些事”。
- 优先级管理分散在工种、目标和个体行为里，不利于做更高层的经营调度。
- 搬运、交付、取料这类中间工作容易暴露为零碎操作，和当前想要降低细节微操的方向不一致。

本设计将玩家控制单位改成“订单”。地图型工作不再停留在裸 `designation`/`blueprint` 层，而是每次操作立即生成一张订单；非地图型生产工作则通过结果单创建。执行层统一只认订单，不再直接认地图标记。

---

## 2. 设计目标

### 2.1 主要目标

1. 建立统一的订单化工作入口，使玩家管理的是“做什么”和“哪件事更重要”，而不是直接驱动每个中间步骤。
2. 地图操作结束后立即生成订单，不需要额外的“二次授权”步骤。
3. 非地图型生产工作仍可通过结果单开单，并进入同一全局执行队列。
4. 殖民者只从全局订单列表中找工作，订单成为默认调度单位。
5. 玩家能够通过统一 UI 查看订单进度、参与者、阻塞原因，并手动调整绝对优先级。
6. 搬运、交付、取料、补料等中间步骤自动派生，不暴露为独立玩家订单。
7. 保持既有架构边界：simulation 持有真实规则和状态，UI 只通过 snapshot/selector 读取并通过 command 发起变更。

### 2.2 非目标

- 本阶段不做自动并单。同类操作重复执行时，默认生成新的独立订单。
- 本阶段不做订单依赖图、批处理优化、团队包工或复杂部门制。
- 本阶段不做“硬指派”或长期专属工作组，只保留软指派语义。
- 本阶段不做跨订单共享子任务的全局最优调度。
- 本阶段不重做全部生产系统细节，只定义它们如何接入订单体系。

---

## 3. 现有架构约束

该设计必须遵守当前项目已经明确的工程边界：

1. gameplay 规则继续放在 simulation 层，UI 不能直接修改 world state。
2. 玩家输入仍然通过命令流写入 simulation，不能由 UI 或输入层直接组装最终 world 结果。
3. `designation`、`blueprint`、区域对象等现有地图投影对象可以保留，但其语义改为“订单项的地图表现”，而不是 AI 直接消费的工作来源。
4. AI 仍然沿用“evaluator -> job -> toil”执行链，只是 evaluator 的工作来源从裸世界对象切换为订单项。
5. 首版应尽量沿用当前输入、UI 和 AI 结构，在现有模块基础上演进，而不是平地起新系统。

---

## 4. 总体方案

本设计采用 **双源开单、统一执行** 模型。

### 4.1 两种开单来源

1. **地图订单**
   - 来源于框选、刷选、拖拽放置等地图操作。
   - 一次操作结束，立即生成一张新订单。
   - 该订单包含本次操作产生的全部目标项。
   - 例：一次框中 5 棵树，生成“1 张砍伐订单 + 5 个目标项”；再次框中 5 棵树，则再生成另一张独立订单。

2. **结果订单**
   - 仅用于非地图框选工作，例如工坊排产、烹饪、加工、制作配方等。
   - 由建筑或面板发起，描述要产出的结果或批次。
   - 最终进入同一全局订单队列，与地图订单一起排序和执行。

### 4.2 统一执行原则

- AI、小人、job evaluator、自动派生子工作统一只认订单。
- 地图上的 `designation`、`blueprint`、区域标记不再是“可以直接取活的工作单位”。
- 玩家统一通过订单看板理解 colony 当前在做什么。

---

## 5. 核心模型

### 5.1 订单层级

系统引入三个明确层级：

1. **WorkOrder（订单）**
   - 玩家可见的主控制单位。
   - 表示一次明确的需求请求。
   - 负责优先级、生命周期、整体进度、暂停/取消、参与者汇总等。

2. **WorkOrderItem（订单项）**
   - 订单中的最小目标单位。
   - 地图订单的订单项通常对应一个目标对象、一块区域或一个蓝图。
   - 结果订单的订单项通常对应一个具体产出批次或配方单元。
   - 小人认领的是订单项或订单项当前可执行阶段，而不是整张订单。

3. **Derived Task（派生步骤）**
   - 系统内部自动拆出的执行步骤，例如取料、搬运、交付、施工、收尾。
   - 只属于执行层，不是玩家直接操作的对象。
   - UI 可显示其阶段信息，但不把它们提升为独立订单。

### 5.2 订单基础字段

首版订单至少需要具备以下概念字段：

- `id`
- `sourceKind`: `map` | `result`
- `orderKind`: build / harvest / mine / cut / zone / craft / cook / process ...
- `status`
- `priorityIndex`
- `createdAtTick`
- `title`
- `items[]`
- `preferredPawnIds[]` 或等价软指派结构
- `summary`
  - `totalItemCount`
  - `doneItemCount`
  - `blockedItemCount`
  - `activeWorkerCount`

### 5.3 订单项基础字段

首版订单项至少需要具备以下概念字段：

- `id`
- `status`
- `targetRef`
  - 地图订单指向地图目标、区域、蓝图或目标集合中的单元
  - 结果订单指向配方、建筑、产出槽位或批次定义
- `currentStage`
  - 例如 waiting / hauling / delivering / building / producing / finishing
- `claimedByPawnId?`
- `blockedReason?`
- `progress?`

### 5.4 状态机

订单层状态：

- `pending`
  - 已创建，尚未有任何项进入执行。
- `active`
  - 至少有一个订单项正在推进，或历史上已经开始推进且尚未终止。
- `paused`
  - 不再分配新工作，但保留数据与地图关联。
- `done`
  - 所有订单项都已完成或合法终止。
- `cancelled`
  - 玩家主动取消，不再参与调度。

订单项层状态：

- `open`
  - 尚未被认领，可进入执行。
- `claimed`
  - 已被某个 pawn 认领，等待实际开始。
- `working`
  - 正在执行派生步骤。
- `blocked`
  - 当前因条件不满足而不可推进。
- `done`
  - 已完成。
- `invalid`
  - 目标已失效，但不应让整张订单永久卡死。

状态规则：

- 一张订单可以同时包含 `working`、`blocked`、`done` 的订单项。
- 只要还有任意可推进项，订单就不算整体阻塞。
- 订单整体显示为阻塞，意味着“当前没有任何可推进项”。

---

## 6. 优先级与调度语义

### 6.1 全局绝对优先级

订单面板采用单一全局列表，不按工种、来源或类别分池。

- 玩家拖动后的列表顺序就是 colony 的默认工作优先顺序。
- 列表越靠前，优先级越高。
- 地图订单和结果订单进入同一条全局序列。

### 6.2 小人取活规则

对于空闲或可切换任务的 pawn，调度器按如下原则工作：

1. 从全局列表顶部开始扫描订单。
2. 在每张订单中寻找“该 pawn 当前能做”的订单项。
3. 找到后立即认领，不继续向下扫描低优先级订单。
4. 如果某张高优先级订单对该 pawn 当前完全不可执行，则继续看下一张订单。

“当前能做”至少要求：

- 订单未暂停、未取消、未完成；
- 订单项处于 `open` 或可重新激活的可执行态；
- 该 pawn 具备工种与角色资格；
- 路径可达；
- 前置条件满足；
- 当前未被其他 pawn 以互斥方式占用。

距离、路径成本、局部效率这类因素只应作为**同一订单内或同一可选集合中的次级排序**，不能压过订单的全局优先级。

### 6.3 软指派

首版保留软指派语义：

- 玩家可以将某个 pawn 软指派到某张订单。
- 被指派的 pawn 在可行时优先尝试该订单。
- 软指派不会压过更高优先级且必须处理的事务。
- 如果该订单当前不可执行，pawn 可以转去执行别的可行工作。

## 7. 玩家交互流

### 7.1 地图订单创建

地图工作采用“操作结束即成单”：

- 框选砍伐、采集、采矿、拆解等：一次框选结束立即生成一张订单。
- 拖拽放置蓝图：一次拖拽放置结束立即生成一张建造订单。
- 区域改造、区域划设、地块改变等：一次刷选/框选结束立即生成一张区域订单。

规则如下：

- 一次操作只生成一张新订单。
- 同类操作不自动并入旧订单。
- 新订单默认插入全局列表末尾。
- 玩家如需加急，之后在订单看板中调整顺序，而不是在地图侧反复补操作。

### 7.2 结果订单创建

非地图工作不通过地图框选，而是通过建筑或面板开结果单：

- 工坊配方排产
- 烹饪批次
- 加工、制造、精炼等

这些结果单与地图订单在表现和排序上应尽量一致，只是创建入口不同。

### 7.3 订单看板

统一订单 UI 的主要职责不是创建，而是**查看、排序和观察执行状态**。

主列表每行代表一张订单，至少展示：

- 订单标题/类型
- 来源类型（地图/结果）
- 总目标数与已完成数
- 当前参与人数
- 是否阻塞
- 当前全局优先级位置
- 暂停/继续/取消状态

看板交互至少包括：

- 拖拽排序
- 暂停
- 继续
- 取消
- 进入订单详情

### 7.4 订单详情

订单详情展示该订单内的目标项信息，至少包括：

- 订单项列表
- 每个目标项当前状态
- 当前执行阶段
- 阻塞原因
- 当前参与的 pawn 列表
- 当前执行者正在处理的步骤概览

### 7.5 地图联动

UI 需要支持订单与地图的双向定位：

- 选中订单时，高亮该订单关联的地图目标。
- 选中地图上的 blueprint / designation / 区域对象时，可以反查所属订单。
- 订单被暂停、取消或完成时，地图表现需要同步更新为正确状态。

## 8. 自动派生子工作

玩家订单只表达目标，不表达执行细节。

例如：

- 建造订单项会自动派生取料、搬运、交付、施工。
- 采集订单项会自动派生靠近目标、执行采集、搬运产出。
- 工坊结果订单会自动派生取料、进工位、生产、产出入库。

设计原则：

1. 派生步骤不进入玩家主订单列表。
2. 派生步骤必须始终挂靠到其父订单项，不能丢失归属。
3. UI 显示阶段信息时，应让玩家看到“这是这张订单中的哪一个目标项，当前处于哪一步”，而不是显示一堆脱离上下文的小任务。
4. 订单排序只影响“下一次认领什么”，不强制打断已经在做的安全原子步骤。

---

## 9. 异常与边界规则

### 9.1 暂停

- 暂停后不再分配新工作。
- 已在执行的小人完成当前安全原子步骤后退出该订单。
- 暂停不清除地图表现与进度。

### 9.2 取消

- 取消后订单进入终止态，不再被调度器扫描。
- 未开始的订单项直接失效。
- 正在执行的工作在安全点终止。
- 对应地图表现应被清理、标记为取消态，或按目标类型转成不可执行态。

### 9.3 目标失效

以下情况不应让系统卡死：

- 树已被别的流程砍掉；
- 蓝图被移除；
- 区域状态已被改变；
- 生产上下文失效；
- 路径条件变化导致局部目标永久不可达。

此时订单项应进入 `invalid` 或等价终止态；整张订单是否结束，取决于剩余项是否全部完成或终止。

### 9.4 阻塞显示

订单项可能因以下原因阻塞：

- 缺料
- 无路
- 缺对应工种
- 前置未完成
- 目标暂不可交互

阻塞信息必须回写到订单项并可在 UI 中查看。  
订单层阻塞为汇总态，表示“当前没有任何可推进项”。

## 10. 与现有模块的关系

本设计不要求立即推翻现有结构，但要求明确各模块职责演进方向：

- `src/adapter/input/input-handler.ts`
  - 继续处理地图输入，但地图操作的最终结果应转向“创建订单及其地图投影”，而不是只创建裸工作标记。

- `src/features/designation/*`
  - 可继续承担地图目标标记创建，但其产物需带有订单归属，语义转为订单项投影。

- `src/features/ai/job-selector.ts`
  - 从“直接扫描地图工作对象”调整为“先扫描订单，再在订单内选可执行项”。

- `src/ui/*`
  - 增加统一订单看板及详情视图。
  - 通过 snapshot/selectors 读取订单状态，继续通过 UI ports / commands 发起排序、暂停、取消、软指派等操作。

---

## 11. 首版范围

首版实现只要求把“订单”建立成默认控制单位和默认调度入口，至少覆盖：

- 地图订单创建
- 结果订单创建
- 全局绝对优先级列表
- 订单级查看与排序
- 订单项级进度、参与者、阻塞展示
- AI 从订单取活
- 中间步骤自动派生
- 暂停 / 继续 / 取消 / 目标失效处理

明确不包含：

- 自动并单
- 订单依赖图
- 跨订单批处理优化
- 硬指派
- 队伍/部门级组织结构

---

## 12. 验证口径

首版验证应围绕玩家可感知行为，而不是只验证内部数据结构：

1. 一次地图操作只生成一张新订单，再次操作同类目标生成另一张独立订单。
2. 地图目标不会在没有订单的前提下被小人直接作为工作来源。
3. 地图订单与结果订单可以出现在同一全局列表中并统一排序。
4. 调整排序后，空闲小人优先消费更高位置的订单。
5. 同一张订单中的多个目标项可以并行推进。
6. 搬运、交付、取料、补料等中间步骤会自动发生，但不会污染玩家主订单列表。
7. 看板能正确展示订单进度、参与者与阻塞原因。
8. 暂停、取消、目标失效等异常路径不会让订单系统卡死。

---

## 13. 设计结论

本方案将玩家操作收敛为两个入口：

- 地图操作直接开地图订单；
- 非地图生产通过结果单开订单；

二者最终都进入一条全局绝对优先级队列，由统一调度器消费。玩家真正控制的是“订单的先后顺序与生命周期”，而不是每个中间步骤或单个裸标记。这样既保留地图操作的直觉性，也把执行层统一成更适合当前项目方向的高层经营接口。
