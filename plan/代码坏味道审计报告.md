## 6. 审计结论总览

本项目当前最突出的坏味道，不是零散命名问题，而是以下几类结构性问题叠加：

1. 高入口、高频变更文件体量过大
2. 单个模块承担多个变化原因
3. 同一业务规则跨多个文件重复表达
4. 状态分支和条件分支持续堆积
5. 核心共享层承载过多领域语义
6. 大量 `any`、字符串协议、布尔值与裸数据承载领域含义
7. 缺少测试保护，导致这些热点区域更不敢重构

## 9. 综合结论

### 9.1 Core / World 方向的总体结论

`src/core` 和 `src/world` 最突出的模式是：

1. 核心共享层承载过多领域语义
2. 同一规则跨多个结构重复表达

前者让改动集中冲击共享文件，后者让通行性、预约等关键规则容易发散和漂移。这两类问题叠加后，会直接放大后续重构和加功能时的修改范围。

### 9.2 Features 方向的总体结论

`src/features` 目前最突出的坏味道，不是单个命名问题，而是“业务编排函数持续膨胀”。

命令处理器、AI 选工、Work 执行器、存档读写都在同一个函数里混合了规则判断、状态迁移和副作用执行，随后又伴随明显的模板化重复。这意味着后续最值得优先治理的方向，不是零碎美化，而是先把高频变更路径上的“长函数 + 重复规则 + 状态分支”拆开，并补上对应行为测试。

### 9.3 Adapter / Main / Presentation 方向的总体结论

`adapter/main/presentation` 的主要问题不是样式层细节，而是：

1. 入口层承担过多组装责任
2. 输入层类体量过大、抽象层混杂
3. 工具切换规则在交互与 UI 层双份维护

这会使交互功能后续演进时，越来越依赖“记住所有同步点”，而不是依赖更清晰的边界设计。

### 9.4 全项目综合判断

这套代码现在最明显的坏味道集中在三条主线上：

1. 入口和交互层过大
2. feature 层大量“规则判断 + 状态迁移 + 副作用”混写
3. 核心层把共享类型和基础协议做得过宽，导致很多业务知识散落到字符串、布尔值和重复分支里

它们会直接带来两个后果：

1. 改一个需求时影响面偏大
2. 团队会越来越不敢动这些热点文件

## 10. 治理优先级建议

基于当前发现，建议的治理顺序如下。

### 10.1 第一优先级

优先治理 3 组高频路径：

1. [`save.commands.ts`](/E:/Me/Twist-v1/src/features/save/save.commands.ts)
2. [`job-selector.ts`](/E:/Me/Twist-v1/src/features/ai/job-selector.ts)
3. [`input-handler.ts`](/E:/Me/Twist-v1/src/adapter/input/input-handler.ts)

原因：

- 体量大
- 职责混
- 改动频繁
- 缺少测试保护
- 重构收益最高

### 10.2 第二优先级

随后处理以下结构性问题：

1. 工具切换逻辑重复
2. 通行性规则重复表达
3. [`core/types.ts`](/E:/Me/Twist-v1/src/core/types.ts) 过载
4. `Designation` / Job 工厂模板化重复
5. 核心协议层 `any` 和字符串协议泛滥

### 10.3 治理策略建议

如果后续要把这份审计结果转成重构任务，建议遵循原始文档中的原则：

1. 在修改需求时顺手清理相关坏味道
2. 优先处理高频变更路径上的坏味道
3. 优先处理会制造缺陷和阻碍测试的坏味道
4. 通过小步重构逐步改善，而不是一次性大手术

同时建议：

1. 没有测试保护时，先补最关键行为测试
2. 每次只做能被验证的小改动
