# 代码坏味道整理

本文基于《重构：改善既有代码的设计》和《代码整洁之道》的核心思想整理，目标不是制造“教条清单”，而是帮助团队更快识别那些会持续拉低可读性、可维护性、可测试性和演化速度的代码问题。

## 1. 什么是“坏味道”

“坏味道”不是编译错误，也不一定会立刻导致线上故障。它更像是一种信号，提示我们：

- 代码的结构可能已经偏离当前业务复杂度
- 局部实现虽然能工作，但正在提高后续修改成本
- 一段代码的意图不够清晰，阅读和修改都需要额外推理
- 模块之间的依赖关系正在变脆

坏味道的危险在于：它们往往不是一次性爆炸，而是让项目在每次迭代里都慢一点、难一点、险一点。

## 2. 两本书对坏味道的共同关注点

《重构》更强调：

- 通过“坏味道”发现重构机会
- 从结构、职责、依赖、重复、数据组织方式来识别设计问题
- 用小步重构降低修改风险

《代码整洁之道》更强调：

- 代码首先是给人读的，其次才是给机器执行的
- 函数、命名、注释、错误处理、类设计、测试都应保持整洁
- “能跑”不等于“整洁”，混乱会不断累积技术债

可以把两本书结合起来理解：

- 《重构》告诉我们“哪里可能有问题”
- 《代码整洁之道》告诉我们“好代码通常长什么样”

## 3. 坏味道总览

从实践角度，可以把常见坏味道分成八类：

1. 体量过大
2. 重复与分散
3. 命名与表达不清
4. 职责混乱与抽象失衡
5. 数据与对象设计失衡
6. 依赖关系脆弱
7. 条件分支与状态控制失控
8. 注释、错误处理与测试失衡

下面按这八类展开。

## 4. 体量过大

### 4.1 过长函数

表现：

- 一个函数做了多个阶段的工作
- 阅读时需要来回滚动才能理解全貌
- 中间变量很多，语义切换频繁
- 条件分支、循环、异常处理混在一起

典型风险：

- 很难命名清楚函数意图
- 修改某一段逻辑时容易影响其他段落
- 单元测试颗粒度很差

改进方向：

- 提炼函数
- 让函数只表达一个清晰意图
- 把“准备数据”“核心决策”“副作用执行”拆开

经验判断：

- 如果一个函数需要靠注释分段，通常已经说明它该拆了
- 如果读者无法用一句话说清“这个函数做什么”，也通常该拆

### 4.2 过大的类

表现：

- 一个类既管状态、又管流程、又管 IO、又管规则
- 字段很多，但每个方法只用其中一部分
- 类名很泛，如 `Manager`、`Processor`、`Service`、`Helper`

典型风险：

- 职责边界模糊
- 改一个场景却影响多个场景
- 复用困难，只能整块复制

改进方向：

- 拆分类职责
- 提炼值对象、策略对象、仓储对象、协调对象
- 按变化原因拆分，而不是只按代码行数拆分

### 4.3 参数列表过长

表现：

- 函数调用时传入很多基础类型参数
- 参数顺序难记，容易传错
- 调用处可读性很差

典型风险：

- 布尔值和数字参数特别容易误用
- 调用方和实现方耦合严重

改进方向：

- 引入参数对象
- 保留真正稳定的最小参数集
- 让参数名体现业务含义，而不是技术细节

## 5. 重复与分散

### 5.1 重复代码

这是两本书都高度警惕的问题。

表现：

- 多处拷贝相似逻辑，只改了少数字段
- 不同模块存在结构相同的条件分支
- 同一业务规则在多个地方重复判断

典型风险：

- 修改规则时容易漏改
- 相同概念在不同地方逐渐漂移
- 测试成本倍增

改进方向：

- 提炼公共函数或模块
- 把重复的“变化点”参数化
- 如果重复背后是不同业务含义，不要硬合并，要先澄清模型

### 5.2 发散式变化

表现：

- 一个需求改动需要同时改很多个文件
- 每次改某个业务规则，都要去多个模块同步修改

典型风险：

- 说明同一个原因导致的变化被散落到很多地方
- 很容易出现局部修复、整体失衡

改进方向：

- 让同一类变化收敛到同一模块
- 把规则聚合到统一抽象下

### 5.3 霰弹式修改

表现：

- 某个小改动引发大量零碎改动
- 每处改动都不大，但涉及文件很多

典型风险：

- 修改成本和心智负担很高
- 回归风险显著增加

改进方向：

- 重新组织职责边界
- 将相关行为和数据放近

## 6. 命名与表达不清

### 6.1 模糊命名

表现：

- `data`、`temp`、`info`、`obj`、`handler2`、`doStuff`
- 类名、函数名只能描述技术动作，不能描述业务意图

典型风险：

- 阅读者只能通过实现猜测意图
- 命名失去导航作用

改进方向：

- 名称要回答“它是什么”“它为什么存在”“它对业务意味着什么”
- 优先使用领域词汇，而不是个人习惯词汇

### 6.2 误导性命名

表现：

- 名称看起来像查询，实际却会修改状态
- 名称看起来像布尔判断，实际返回复杂对象
- 名称叫 `list`，实际是字典；叫 `save`，实际还会触发通知和缓存清理

典型风险：

- 调用方基于错误预期使用接口
- 隐式副作用增多

改进方向：

- 让名称与行为一致
- 查询和命令分离

### 6.3 代码需要注释才能懂

表现：

- 注释在解释“这段代码其实是在做什么”
- 删掉注释后代码几乎不可读

典型风险：

- 说明代码表达力不足
- 注释后续常与代码脱节

改进方向：

- 优先改进命名、拆分函数、提炼变量
- 注释应解释“为什么”，不是重复“做了什么”

## 7. 职责混乱与抽象失衡

### 7.1 一个模块做太多层次的事

表现：

- 同一函数里同时出现业务规则、数据库访问、UI 拼装、日志输出
- 抽象层次频繁跳变

典型风险：

- 难以测试
- 复用困难
- 修改一个层次的问题时会牵动其他层次

改进方向：

- 分离业务决策与基础设施细节
- 同一个函数尽量保持抽象层次一致

### 7.2 违背单一职责

表现：

- 一个模块有多个变化原因
- 类接口很大，但调用方只使用其中一部分

典型风险：

- 一个修改理由会破坏另一个修改理由
- 团队协作冲突增多

改进方向：

- 识别“变化原因”
- 以变化维度而非代码位置划分职责

### 7.3 不恰当的抽象

表现：

- 过早抽象：为了“将来可能复用”而引入复杂层次
- 过度通用：接口看似灵活，实际谁都看不懂
- 错误继承：为了复用代码强行建立 is-a 关系

典型风险：

- 抽象反而遮蔽真实意图
- 新需求稍一变化，抽象就崩

改进方向：

- 先消除重复，再抽象
- 优先组合而非滥用继承
- 抽象应服务当前稳定模式，而不是想象中的未来

### 7.4 中间人

表现：

- 某个类大部分方法只是简单转发到另一个对象

典型风险：

- 增加理解层次，却没有增加语义价值

改进方向：

- 去掉无意义代理
- 只保留真正形成边界、隔离变化或补充语义的包装层

## 8. 数据与对象设计失衡

### 8.1 基本类型偏执

表现：

- 大量使用裸字符串、数字、布尔值表示业务概念
- 如用户 ID、金额、坐标、状态码都直接用基础类型

典型风险：

- 约束散落各处
- 参数易混淆
- 无法在类型层面表达领域语义

改进方向：

- 引入值对象
- 把校验、格式化、比较规则收进对象

### 8.2 数据泥团

表现：

- 一组字段总是一起出现
- 多个函数总是同时传递同样几项数据

典型风险：

- 说明数据之间有稳定关系，但模型没有显式表达出来

改进方向：

- 提炼对象
- 让数据组合对应真实概念

### 8.3 被拒绝的遗赠

表现：

- 子类继承了父类，却不真正需要其大部分行为
- 甚至需要重写或禁用父类方法

典型风险：

- 继承关系失真
- 子类与父类语义不一致

改进方向：

- 重新审视继承关系
- 优先组合、委托、接口分离

### 8.4 临时字段

表现：

- 某些字段只在特定场景下才有意义
- 对象大部分时间处于“字段为空但必须存在”的状态

典型风险：

- 对象状态空间混乱
- 调用方不得不处处判空

改进方向：

- 提炼专用对象
- 用状态对象或更明确的生命周期模型替代

### 8.5 过度暴露数据

表现：

- 对象只是字段容器，外部随意读取和修改
- 行为散落在对象外部

典型风险：

- 数据和行为分离，规则难以收口
- 不变量难以保证

改进方向：

- 把核心规则放回对象
- 让对象负责维护自己的合法状态

## 9. 依赖关系脆弱

### 9.1 依恋情结

表现：

- 一个函数频繁操作另一个对象的数据，却不在那个对象里
- 某个模块“比自己更了解别人”

典型风险：

- 说明行为放错了位置
- 数据和行为分离造成高耦合

改进方向：

- 把行为移动到真正拥有数据的对象

### 9.2 循环依赖

表现：

- 模块 A 依赖 B，B 又反过来依赖 A
- 类与类之间互相知道太多

典型风险：

- 初始化复杂
- 难以测试与替换
- 架构边界逐渐失效

改进方向：

- 提炼接口
- 引入中介、事件、回调或更清晰的分层

### 9.3 不必要的耦合

表现：

- 一个简单功能必须依赖庞大上下文才能运行
- 工具函数偷偷依赖全局状态

典型风险：

- 复用困难
- 测试必须搭很大环境

改进方向：

- 缩小依赖面
- 明确输入输出
- 尽量减少隐藏依赖

## 10. 条件分支与状态控制失控

### 10.1 过多的条件分支

表现：

- `if/else` 或 `switch` 很长
- 相同判断逻辑在多处出现
- 每加一个类型或状态就要改多个分支

典型风险：

- 扩展成本高
- 漏改风险高

改进方向：

- 用多态、策略、映射表、状态模式替代部分条件分支
- 把分支背后的变化轴识别出来

### 10.2 以状态码驱动的分散逻辑

表现：

- 模块各处都在判断 `"NEW"`、`"RUNNING"`、`"DONE"`、`"FAILED"`
- 状态迁移规则散落在不同函数里

典型风险：

- 状态机不可见
- 非法状态迁移容易出现

改进方向：

- 收敛状态流转逻辑
- 用显式状态模型表达允许的迁移

### 10.3 控制标记与布尔参数滥用

表现：

- 函数参数里有多个布尔值，如 `save(user, true, false, true)`
- 一个方法根据布尔参数走完全不同路径

典型风险：

- 可读性极差
- 一个函数承担多个职责

改进方向：

- 拆成多个有语义的函数
- 让调用动作直接表达意图

## 11. 注释、错误处理与测试失衡

### 11.1 赘余、过时或误导性注释

表现：

- 注释重复代码表面含义
- 代码已改，注释没改
- 用大段注释掩盖糟糕结构

典型风险：

- 注释成为新的噪音来源
- 读者被错误信息误导

改进方向：

- 删除废话注释
- 保留少量解释动机、约束、取舍的注释

### 11.2 错误处理污染主流程

表现：

- 主逻辑被大量判空、错误码分支、异常兜底打断
- 一层层传播错误，却没有统一语义

典型风险：

- 主流程难读
- 异常语义不一致

改进方向：

- 分离正常路径与错误路径
- 建立统一错误模型
- 在边界层做转换和兜底

### 11.3 缺乏测试或测试不可维护

表现：

- 没有测试，或测试只能覆盖表面 happy path
- 测试高度依赖实现细节
- 改代码时测试成片失效，却无法说明真正行为是否变了

典型风险：

- 不敢重构
- 测试不能提供真实反馈

改进方向：

- 测试行为，不要过度绑定内部实现
- 优先补关键业务规则、边界条件、回归场景

《代码整洁之道》尤其强调：脏代码和脏测试会相互强化，最终让团队既不敢改产品代码，也不相信测试结果。

## 12. 从“症状”快速识别坏味道

如果团队在日常开发中经常出现下面这些感受，往往说明坏味道已经比较明显：

- “这段代码我不敢动，一动就容易出事”
- “同样的规则到底改哪儿，得全局搜一下”
- “函数名字看不出来在干嘛，只能进去读实现”
- “一个需求改动涉及十几个文件，但每个文件只改一点点”
- “测试很多，但一改结构就全红”
- “每次新增一种类型，都要去复制一套 if/else”
- “接口参数太多，不看定义不敢调用”
- “注释比代码更长，但还是看不懂”

这些感觉本身就是重要信号。

## 13. 坏味道不等于必须立刻重写

识别坏味道的目的是指导重构，而不是引发大规模推倒重来。更合理的做法是：

1. 在修改需求时顺手清理相关坏味道
2. 优先处理高频变更路径上的坏味道
3. 优先处理会制造缺陷和阻碍测试的坏味道
4. 通过小步重构逐步改善，而不是一次性“大手术”

重构时应遵循两个原则：

- 没有测试保护时，先补最关键的行为测试
- 每次只做能被验证的小改动

## 14. 一个实用的判断标准

当你怀疑一段代码有坏味道时，可以快速问自己六个问题：

1. 这段代码是否容易读懂，而不是勉强读通？
2. 这段代码的名字、结构和行为是否一致？
3. 修改一个需求时，影响范围是否合理可控？
4. 相同知识是否只在一个地方表达？
5. 模块边界是否清楚，依赖是否自然？
6. 我们是否能低风险地为它写测试、改逻辑、做重构？

如果这六个问题里有多个答案是否定的，就很可能已经存在明显坏味道。

## 15. 团队治理建议

为了避免坏味道持续堆积，可以在团队里形成下面这些默认习惯：

- 代码评审时不仅看“能不能跑”，也看命名、职责、重复和依赖
- 新增功能时优先复用领域概念，避免继续堆裸数据和特殊分支
- 每次需求开发顺手清理触达范围内的小坏味道
- 对高频修改区域定期做专题重构
- 用测试保护重构，而不是靠记忆和勇气重构
- 把“代码可读性”作为交付质量的一部分

## 16. 结语

坏味道的本质，不是代码“难看”，而是代码开始阻碍团队理解系统和安全修改系统。  
《重构》让我们学会发现结构性问题，《代码整洁之道》提醒我们持续维护表达质量。把两者结合起来，团队就能更早发现风险、更稳地演进代码，而不是在复杂度失控后被迫返工。
