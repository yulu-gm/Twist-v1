## 二、职责过重的类/文件

### 2.1 toil-executor.ts（677 行）— 最严重

**问题：** 一个文件承担了 toil 状态机调度 + 7 种 toil 的具体执行逻辑 + 任务完成/中断处理。

当前结构：
```
executeToil()
  ├─ GoTo      — 寻路 + 移动进度
  ├─ PickUp    — 物品拾取 + 对象销毁
  ├─ Drop      — 物品投放 + 对象创建
  ├─ Work      — 进度推进 + 指派完成
  ├─ Wait      — 等待 + 饥饿恢复
  ├─ Deliver   — 多阶段投递 + 蓝图追踪
  └─ Interact  — 位置校验 + 交互
completeJob()  — 清理/事件/预约释放
interruptJob() — 中断清理
```

**坏味道：**
- `executeToil()` 内 switch-case 7 分支，每个分支 30-80 行
- 直接操作对象的 `defId`/`stackCount`/`growthProgress`，与 feature schema 紧耦合
- 18 处 `as any` 类型强转

**建议拆分：**
```
features/ai/
├─ toil-executor.ts          ← 仅保留状态机调度 (~80 行)
├─ toil-handlers/
│   ├─ goto.handler.ts
│   ├─ pickup.handler.ts
│   ├─ drop.handler.ts
│   ├─ work.handler.ts
│   ├─ wait.handler.ts
│   ├─ deliver.handler.ts
│   └─ interact.handler.ts
├─ job-completion.ts         ← 任务完成逻辑
└─ job-interruption.ts       ← 任务中断逻辑
```

### 2.2 game-map.ts（423 行）— 神容器

**问题：** 一个文件内定义了四个独立子系统，只因为它们都挂在 GameMap 上。

**建议拆分：**
```
world/
├─ game-map.ts               ← 仅 GameMap 容器 (~100 行)
├─ zone-manager.ts           ← 玩家区域管理
├─ room-graph.ts             ← 房间连通性
├─ reservation-table.ts      ← 资源预约
└─ path-grid.ts              ← 可行走性网格
```

### 2.3 dom-ui-manager.ts（424 行）— UI 混合状态同步

**问题：** DOM 元素缓存 + 事件绑定 + 5 个脏检查同步方法（`prevClock`/`prevSpeed`/`prevTick`/`prevPawnCount` 缓存）。

**建议：** 将脏检查状态同步提取到单独的 `UIStateSync` 类，或引入轻量响应式机制。

## 三、类型安全问题（32 处 `as any`）

### 分布

| 模块 | 数量 | 根因 |
|------|------|------|
| `features/ai/*` | 14 | Pawn 接口鸭子类型、直接访问 feature 属性 |
| `adapter/render/*` | 7 | 渲染器需要访问 feature 专属字段 |
| `features/construction/*` | 3 | 访问 ConstructionSite 的专属属性 |
| 其他 | 8 | 分散在 designation、debug、inspector |

### 根因分析

`core/map-object.ts` 定义了一个巨大的联合类型：

```typescript
export type MapObject = Pawn | Building | Item | Plant | Fire | Corpse | Blueprint | ConstructionSite | Designation;
```

这导致 **core 层反向依赖 features 层**，同时 features 层操作对象时无法安全地窄化类型，被迫用 `as any`。

### 改进方向

- `MapObjectBase` 保留在 core，作为所有对象共享的接口
- 各 feature 类型不再聚合到 core 的联合类型
- 使用 `kind` 字段的 discriminated union + 类型守卫函数取代 `as any`：

```typescript
function isPawn(obj: MapObjectBase): obj is Pawn {
  return obj.kind === ObjectKind.Pawn;
}
```

## 四、重复代码

### 4.1 物品创建逻辑（5 处重复）

在以下位置存在几乎相同的 "创建掉落物品" 代码：

- `toil-executor.ts` — Drop toil
- `toil-executor.ts` — 挖矿产出
- `toil-executor.ts` — 收割产出
- `cleanup.ts` — 清理掉落
- `construction.commands.ts` — 蓝图材料退还

**共同模式：**
```typescript
const item = {
  id: nextObjectId(),
  kind: ObjectKind.Item,
  defId, mapId: map.id, cell,
  tags: new Set(['haulable', 'resource']),
  destroyed: false,
  stackCount: count,

map.objects.add(item as any);
```

**建议：** 统一使用 `item.factory.ts` 中的工厂函数。

`toil-executor.ts`、`job-selector.ts`、`cleanup.ts` 各自定义了一套 Pawn 的接口片段。

**建议：** 在 `features/pawn/pawn.interface.ts` 定义共享接口。

## 五、扩展性瓶颈

### 5.1 新增对象类型需修改 3 个核心文件

1. `core/types.ts` — 添加 ObjectKind 枚举值
2. `core/map-object.ts` — 添加到 MapObject 联合
3. `render-sync.ts` — 注册渲染器

**改善方向：** 对象类型注册表模式，新类型只需在自己的 feature 目录下注册。

### 5.2 新增任务类型需修改 AI 核心

1. `job-selector.ts` — 在 `gatherCandidates()` 中添加新候选
2. `toil-executor.ts` — 如有新 toil 类型需修改 switch

**改善方向：** 任务工厂注册表 + toil 处理器注册表，解耦 AI 框架与具体任务。

### 5.3 新增指派类型需修改核心枚举

`DesignationType` 枚举在 `core/types.ts` 中，新增类型无法避免修改核心。

## 七、优先级建议

### P0 — 不处理会阻碍后续开发

1. **拆分 toil-executor.ts** → toil handler 注册表模式
2. **拆分 game-map.ts** → 子系统各自独立文件
3. **消除核心类型反向依赖** → 移除 `core/map-object.ts` 对 features 的 import

### P1 — 不处理会增加维护成本

4. **统一物品创建** → 复用 `item.factory.ts`
5. **合并 Pawn 鸭子类型** → 共享接口
6. **拆分 dom-ui-manager.ts** → 分离状态同步

### P2 — 有余力时改善

7. **注册表模式** — toil handler / job factory / designation type
8. **拆分 input-handler.ts 的 setupInputs()** — 按输入类型分组
