# 001-实体系统核心模型落地计划

## 1. 计划目标
根据 `oh-code-design/实体系统.yaml` 的设计，在代码中落地实体系统（Entity System）的核心架构。当前项目中的实体（如小人、物资、树木、岩石）散落在 `src/data/` 和 `src/game/` 的多个文件中（例如 `pawn-profiles.ts`、`ground-items.ts`），缺乏统一的生命周期管理和唯一标识。本计划旨在建立统一的实体目录和实体原型定义，为后续工作系统、地图系统和行为系统的重构提供统一的领域对象基础。

## 2. 工作内容
1. **定义实体基础类型**：
   - 在 `src/game/` 下新增实体系统核心文件（如 `entity-system.ts`）。
   - 定义全局唯一的实体标识类型（Entity ID）。
   - 严格按照 `oh-code-design/实体系统.yaml` 中的“核心数据”定义 `小人实体`、`物资实体`、`树木实体`、`岩石实体`、`建筑蓝图实体`、`建筑实体`、`区域实体` 的数据结构。

2. **实现实体目录 (Entity Directory)**：
   - 实现集中管理所有实体的状态容器（如 `EntityRegistry` 或 `EntityManager`）。
   - 提供按实体标识（ID）、实体类型（Type）、位置（Position）的查询接口。
   - 提供实体的创建、删除、更新等基础状态变更方法。

3. **实现基础生命周期与关系规则骨架**：
   - 建立实体创建和销毁的基础规则层（预留具体转化逻辑的扩展点，如树木转木头）。
   - 建立基础的一致性校验逻辑骨架（如位置与归属的互斥关系）。

4. **迁移现有 Mock 数据至实体系统**：
   - 将 `src/data/ground-items.ts` 中的散落物资和 `src/data/pawn-profiles.ts` 中的小人档案数据，转化为标准的实体对象并注入到实体目录中。
   - 调整现有的读取逻辑，使 UI 和场景渲染从新的实体目录中获取数据，从而移除部分硬编码的 mock 字典。

## 3. 验收标准
- `src/game/` 中存在明确的实体系统核心类型定义与目录管理代码。
- 实体类型定义完全覆盖 `oh-code-design/实体系统.yaml` 中列出的所有实体类型及关键字段。
- 现有的 mock 物资和小人数据成功迁移至统一的实体目录管理，且移除原有的硬编码数据源。
- 项目执行 `npm run build` 编译通过。
- 游戏场景能正常渲染小人和物资，且数据流向清晰（UI -> 实体系统只读投影）。
- 本计划保持话题单一，不涉及复杂的工作调度或行为树逻辑，仅聚焦于实体数据模型的统一。
