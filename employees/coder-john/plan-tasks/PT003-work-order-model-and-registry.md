# PT003-工作单模型与工作目录骨架计划

## 1. 计划目标

依据 `oh-code-design/工作系统.yaml` 中的 **工作模型层** 与模块 **「工作目录」**，在代码中落地与实现无关的 **工作单**、**工作步骤**、**工作占用记录** 数据结构，以及集中保存与查询用的 **工作目录**（内存容器即可）。

本计划话题单一：**只建立工作领域数据与目录访问接口**，不实现工作生成器、编排器、领取器、结算器，也不改行为系统或直接把 `EntityLifecycle` 接入游戏主循环（此类接线属于后续独立计划）。

## 2. 工作内容

1. **类型定义（对齐 YAML「核心数据」）**  
   - **工作单**：至少包含 YAML 所列关键字段——工作标识、工作类型、工作状态、目标实体标识、目标位置、发起原因、优先级；类型与字段命名遵循现有 `src/game/` 风格（英文字段名 + 与设计语义对应的注释可有可无，禁止过程性说明注释）。  
   - **工作步骤**：步骤类型、执行前置条件、成功结果、失败结果（若当前无对应枚举，可用 `string` 或最小联合类型占位，但含义须与文档一致）。  
   - **工作占用记录**：工作标识、小人标识、锁定目标、锁定时间（时间可用 `number` 毫秒或项目已有时间抽象，与现有代码一致即可）。

2. **工作目录（`WorkRegistry` 或等价命名）**  
   - 按工作标识增删改查；保留「待执行、执行中、完成、失败」等状态分区或可按状态过滤的查询。  
   - 提供按工作类型、优先级、目标位置（或与 `GridCoord` 可比较的键）的查询入口，满足 YAML「支持按类型、优先级、位置查询」的表述即可，无需优化为空间索引。  
   - 工作占用记录可内嵌于工作单或独立映射，须保证「同一锁定目标与工作」可被检索，便于后续领取器实现。

3. **文件与依赖**  
   - 新增文件置于 `src/game/`（例如 `work-system.ts`），仅依赖已有坐标/实体标识类型（如 `EntityId`、`GridCoord`）若需要；**禁止**在本计划中引入对 `GameScene` 或 Phaser 的依赖。

## 3. 验收标准

- `src/game/` 中存在工作单相关类型与工作目录实现，字段覆盖 `工作系统.yaml` 中「核心数据」三节所述关键信息。  
- 工作目录支持注册工作、更新状态、按状态/类型/优先级/位置做基础查询，API 含义清晰。  
- `npm run build` 编译通过。  
- 与 `PT001`、`PT002` 无职责重叠：本计划不修改实体原型与 `EntityLifecycle` 行为，除非为消除编译错误所必需的极少量类型导出调整（应尽量避免）。
