# 静态实体视图同步缺陷分析与重构报告

**撰写人**: Coder John
**日期**: 2026-04-07
**相关模块**: `GameScene.ts`, `renderers/*`, `EntityRegistry`

## 1. 现象与问题描述

在当前的游戏版本中，当小人（Pawn）完成“开采岩石”或“砍伐树木”的工作后，底层数据（`EntityRegistry`）已经正确地移除了对应的岩石/树木实体，并在对应的坐标生成了掉落物（如“石块”、“木柴”）。

**缺陷表现**：游戏画面完全没有更新。被挖掉的石头依然显示在原地，新生成的掉落物也没有渲染出来。玩家在视觉上无法感知到工作已经完成。

## 2. 根本原因分析

该问题是由**代码实现层面的双标**与**Agent工作模式的盲区**共同导致的：

### 2.1 渲染管线的“双标”实现
*   **动态实体（Pawn）**：实现了严格的视图同步机制。在 `GameScene.update()` 中，每帧调用 `syncPawnViews`，通过比对数据状态来更新小人的位置、动画和状态文本。
*   **静态实体（Rock, Tree, GroundItem, Building）**：开发者（或之前的 Agent）潜意识认为它们是“静态”的，为了图省事，直接在 `GameScene.create()` 中调用了 `drawStoneCells`、`drawGroundItemStacks` 进行了一次性绘制（Fire-and-Forget）。
*   **结果**：当静态实体的状态发生变化（被销毁、数量叠加）时，没有任何机制去通知 Phaser 渲染层进行重绘或销毁。

### 2.2 Agent 编程的视觉盲区
Agent 在验证功能时，主要依赖于读取内存数据或运行单元测试。当 Agent 查阅 `EntityRegistry` 发现岩石数据已被删除、掉落物数据已生成时，便判定“任务圆满完成”。由于缺乏端到端的视觉多模态验证，Agent 持续忽略了“画面未刷新”这一对人类而言极其明显的低级错误。

## 3. 架构对齐与合规性检查

查阅 `oh-code-design/` 目录下的架构设计：
*   **《实体系统.yaml》** 明确指出实体系统需要“为 UI、交互反馈、调试视图提供只读投影”。
*   **《UI系统.yaml》** 明确指出地图反馈层需要“刷新选区、高亮、虚影、标记、进度条”。

**结论**：当前静态实体的一次性绘制实现，**违背了架构设计中“数据作为唯一事实来源，视图按需刷新投影”的理念**。

## 4. 重构方案与执行计划（统一渲染器注册表模式）

为了彻底解决该问题，并避免未来新增实体类型时产生“面条代码”或遗漏同步逻辑，决定采用**统一视图映射表 + 渲染器注册机制**进行重构。

### 4.1 定义泛型渲染器接口
在 `src/scenes/renderers/` 下引入统一的接口规范，约束所有实体的渲染生命周期：
```typescript
export interface EntityRenderer<TEntity extends AnyGameEntity, TView> {
  shouldRender?(entity: TEntity): boolean;
  create(scene: Phaser.Scene, entity: TEntity, grid: WorldGridConfig, ox: number, oy: number): TView;
  update(scene: Phaser.Scene, entity: TEntity, view: TView, grid: WorldGridConfig, ox: number, oy: number): void;
  destroy?(view: TView): void;
}
```

### 4.2 建立统一视图映射表与分发器
在 `GameScene` 中废弃分散的 Map，仅保留一个全局的视图映射表：
```typescript
private entityViews = new Map<EntityId, Phaser.GameObjects.GameObject | Phaser.GameObjects.Container>();
```
实现一个核心分发器 `syncEntityViews`：
1.  遍历所有注册了渲染器的实体类型。
2.  **Create (新增)**：若实体 ID 不在 View Map 中，调用对应渲染器的 `create` 实例化 Phaser 对象并存入 Map。
3.  **Update (更新)**：若实体 ID 已存在，调用对应渲染器的 `update` 刷新显示属性。
4.  **Destroy (销毁)**：遍历 View Map，若 ID 在 `registry` 中已不存在（或不再满足 `shouldRender`），则调用 `destroy()` 清理内存并从 Map 移除。

### 4.3 重构现有 Renderers
将现有的 `drawXXX` 逻辑封装为具体的 Renderer 实现：
*   `RockRenderer` / `TreeRenderer`：实现 `create` 绘制静态图形，`update` 留空。
*   `GroundItemRenderer`：实现 `shouldRender`（仅渲染 `containerKind === "map"` 的物资），实现 `create` 绘制框和文字，实现 `update` 刷新数量文本。
*   `BuildingRenderer`：实现 `create` 绘制建筑占位图和标签。

### 4.4 挂载主循环
在 `GameScene.update()` 的最后，统一调用同步管线：
```typescript
syncEntityViews(
  this,
  this.entityViews,
  this.entityRegistry,
  this.worldGrid,
  this.gridOriginX,
  this.gridOriginY,
  this.renderers // 包含所有具体 Renderer 的字典
);
```

## 5. 后续防呆建议

1.  在 `oh-code-design/UI系统.yaml` 中增加强制约束：“所有场景实体（含静态环境与掉落物）的渲染必须基于 EntityId 映射表进行生命周期同步（创建/更新/销毁），严禁在初始化时进行一次性静态绘制。”
2.  未来新增任何实体类型（如陷阱、动物、特效），只需实现对应的 `EntityRenderer` 并注册到字典中即可，主循环代码实现 **开闭原则 (OCP)**，无需任何修改。
